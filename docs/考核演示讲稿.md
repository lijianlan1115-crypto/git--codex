# 考核演示讲稿

## 开场

我的项目题目是《基于Codex的研发协同变更分析与测试预警工作流》。它解决的不是单点 Bug 修复，而是多人协作开发中“需求变了、代码改了、同事提交了以后，谁会被影响、应该怎么验证”的问题。

## 场景一：我的变更协同

我准备修改或刚修改完代码时，系统读取本地 Git 代码库的 `git diff`，调用 Codex 理解代码变化，再结合 V27 知识库输出影响范围、风险等级和测试方案。

演示命令：

```bash
python3 scripts/build_change_summary.py \
  --repo hotel--ota-ai \
  --requirement "分析当前本地修改是否影响价格、库存和收益策略"
```

展示 `outputs/change_summary-latest.md`。

## 场景二：需求变更后自动修改

当需求发生变化时，我可以让 Codex 进入本地 Git 代码库进行代码修改。修改后系统再读取实际 `git diff`，生成协同影响和测试预警。

演示命令：

```bash
python3 scripts/apply_requirement_change.py \
  --repo hotel--ota-ai \
  --requirement "在价格策略中增加价格下限校验，并生成测试预警"
```

展示：

- 本地 Git 文件发生变化。
- `outputs/codex_patch.diff`
- `outputs/change_summary-latest.md`

## 场景三：HTTP 工作流集成

如果接扣子或飞书，外部平台只需要调用 HTTP 接口。扣子不是核心能力，只是工作流入口和展示层。

```bash
curl -X POST http://127.0.0.1:8000/analyze-change \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer test-token-123" \
  -d '{"repo":"hotel--ota-ai","requirement":"分析当前本地修改的协同风险"}'
```

## 总结

这个项目的核心价值是把“开发凭经验判断影响”变成“AI 基于本地 Git 代码库、实际 diff 和 V27 业务知识库自动生成协同预警”。它能降低遗漏、返工和沟通成本，提高开发与测试协同效率。
