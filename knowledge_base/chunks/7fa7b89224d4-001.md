# S16 进度偏差诊断引擎

## Reference 导航

执行本 skill 前先读取本目录 `references/` 五件套：

- `references/input_schema.json`：输入字段和字段质量。
- `references/output_schema.json`：统一输出结构。
- `references/rules.md`：多源依据、判断逻辑、异常和安全规则。
- `references/examples.md`：飞书话术、样例输入、runtime 输出和最终回复。
- `references/runtime_commands.md`：可调用的稳定脚本命令。

## 适用场景

当用户询问今天销售进度是否正常、为什么落后/领先、当前应该做什么动作时，使用本 skill。

## 必须读取

- `{baseDir}/../_shared/common-contract.md`

## V27 架构绑定

- 本 skill 的节点、Agent、上下游和禁止动作以 `references/v27_alignment.json` 为准。
- 架构事实源只引用 `architecture/node_registry.json`、`architecture/edge_registry.json`、`architecture/scenario_chain_registry.json`。
- 字段事实源只引用 `contracts/field_registry.yaml` 和 `contracts/node_io_contract.yaml`，不要复制全量字段池。
- Demo 输入使用 `examples/demo_data/nodes/N011.json`。
- Demo 输出必须保留 `data_source_type=demo_data`，并标注 `preview_only` 或 `dry_run`。
- 若上述绑定文件缺失，必须输出 `config_gap`，不得把缺失绑定伪装成 confirmed。

## 核心职责

- 调用 runtime `deviation` 生成当前代码实际支持的 `direction`、`actual_orders`、`target_orders`、`progress_checkpoint`、`checkpoint_target_orders` 和 `checkpoint_completion_rate`。
- 优先按 S15 节点目标判断进度，不只看全天 `daily_completion_rate`。
- 校验 `actual_source`、`target_source`、`actual_basis_date`、`target_basis_date` 和 `field_freshness`，判断是否允许继续交给 S5。
- 先区分流量不足和转化不足；流量不足优先转 S9/S14，转化不足且数据 fresh 时才允许进入 S5 候选。
- S16 只输出诊断信号和下游建议，不直接执行调价、推广、评论或房量动作。

## 执行流程

1. 确认 `hotel_id`、营业日期、用户权限和数据新鲜度。
2. 读取最新 S2 经营快照、S15 基准线、订单/预订明细和流量转化证据。
3. 执行：

```bash
python runtime/hotel_ota_runtime.py deviation --hotel-id puyue
python runtime/hotel_ota_runtime.py normalize-sample --sample meituan-room-count
python runtime/hotel_ota_runtime.py normalize-sample --sample dindanll-order
```

4. 判断状态：`ahead`、`normal`、`behind`、`data_gap` 或异常。
5. 当 `downstream_allowed=false` 时，只做历史/演示诊断，不触发 S5。
6. 只给一到两个最高优先级动作，避免建议过多。

## 输出要求

返回进度状态、节点完成率、全天完成率、实际来源、目标来源、字段新鲜度、阻断原因、归因、建议、下游 skill、下一次检查时间。

## 安全规则

- 不直接执行调价或推广。
- 归因不确定时必须说明“需补数据”。
- demo/sample/stale/missing_date 只能用于预览、dry-run 或历史复盘，不能触发正式调价链路。
