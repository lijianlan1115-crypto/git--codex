# 前台执行清单算法

## 1. 功能名称

前台执行清单生成。

## 2. 目标

根据需求指数、库存、OTA健康、评论问题和店长策略，自动生成前台班次执行清单。

前台不是决策者，而是执行终端。系统需要把店长和算法的策略转成可执行任务。

## 3. 输入数据

- 当前日期和班次。
- 当前需求指数。
- 当前OTA健康分。
- 当前各房型库存。
- 当前各房型价格。
- 当前曝光、浏览、转化数据。
- 当前评论异常。
- 当日中高价房型入住客人。
- 当日目标好评房间。
- 店长策略模式。
- 是否有特殊活动日。

## 4. 核心指标

```text
库存压力 = 1 - 房型剩余可售比例
高流低转 = 曝光/浏览高于历史，支付转化低于历史
低流高转 = 曝光/浏览低于历史，支付转化高于历史
目标好评客人 = 中高价房型客人 + 沟通积极客人 + 服务体验较好客人
```

## 5. 课程依据

课程第17条给出前台八项日常工作：

- 上岗交接。
- 查房态数据。
- 查外网。
- 回复点评和特殊点评上报。
- 查阅值班日志。
- 办理入住和离店。
- 间隔2小时通报。
- 撰写值班日志和交接班。

课程强调：

- 9点开始每2小时报一次大床/双床库存剩余。
- 结合库存确定当天流量态度和价格态度。
- 实时曝光浏览转化率数据截图发群。
- 店长据此确定策略。

## 6. 任务生成逻辑

### 6.1 固定任务

每个班次都生成：

- 上岗交接。
- 核对备用金和登记表。
- 查询房态。
- 查询外网。
- 回复点评。
- 查阅值班日志。
- 记录特殊事项。
- 交班总结。

### 6.2 每2小时任务

每2小时生成：

- 上报各房型库存。
- 上报曝光、浏览、支付转化。
- 上报是否高流低转或低流高转。
- 检查引流房是否正确。
- 检查房型排列顺序。
- 检查价格梯队价差。

### 6.3 需求低谷任务

触发条件：

```text
需求指数 <= 25
```

任务：

- 检查引流房是否可售。
- 检查是否有房型价差过大。
- 检查取消政策是否过严。
- 重点反馈库存压力。
- 提醒店长是否需要活动或降价评估。

### 6.4 旺日/爆发任务

触发条件：

```text
需求指数 >= 51
```

任务：

- 高频检查库存。
- 标记低库存房型。
- 检查是否过早卖完主力房型。
- 重点跟进中高价房型客人。
- 避免错误承诺低价或随意升级。

### 6.5 评论异常任务

触发条件：

```text
出现差评或高风险评论
OR 同类问题连续出现
```

任务：

- 特殊点评上报店长。
- 标记相关房间和责任环节。
- 后续入住客人重点规避同类问题。
- 进入整改清单。

## 8. 可配置参数

- 班次时间。
- 每2小时通报时间点。
- 哪些指标必须截图。
- 目标好评客人规则。
- 需要店长升级的任务类型。
- 各房型库存上报格式。
- 是否按电竞酒店特有设备检查生成任务。

## 9. 数据列表适配校验

### 9.1 可直接使用的数据

来自 `数据列表.txt` 的可用字段：

- 今日预抵。

- 房态类型。

- 操作员/操作人。
- 操作时间。
- 创建人、审批人。

### 9.2 算法修正

前台执行清单不应只生成“提醒”，还应支持任务闭环。

新增任务状态：

```text
待执行
执行中
已完成
需店长确认
异常挂起
```

新增任务完成校验：

```text
库存上报任务 -> 必须有操作人、操作时间、库存数据
好评任务 -> 对应好评完成量或评价新增
推广任务 -> 对应推广任务进度
差评任务 -> 对应差评跟进状态
异常任务 -> 对应通知日志或店长确认
```

### 9.3 逐字稿校验依据

逐字稿第17条确认：前台按八项日常工作执行，每2小时通报库存、后台数据和外网检查。

逐字稿第17条确认：前台要回复点评，特殊点评及时上报团队周知。

逐字稿第17条确认：中高价房型和目标好评客人需要写入值班日志并精准跟进。

### 9.4 准确性与效率结论

该算法应从“生成清单”升级为“生成清单 + 跟踪完成”。数据列表已经具备任务执行闭环字段，适合做成可追责的 SOP 系统。
