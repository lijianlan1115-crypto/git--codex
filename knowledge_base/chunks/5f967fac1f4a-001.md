## 1. 总体原则

本蓝图不是开发文档的最终版，而是酒店运营自动建议系统的第一版功能算法拆解。

每个功能独立计算，独立输出建议，最后由总分析层整合。

统一功能结构：

```text
功能名称
目标
输入数据
核心指标
判断逻辑
计算方法
输出结果
异常处理
可配置参数
```

## 3. 功能依赖关系

建议把“需求指数”作为底层公共变量。

```text
需求指数
 -> 调价建议
 -> OTA活动建议
 -> 前台执行清单
 -> 收益最大化方案

OTA健康诊断
 -> 流量转化诊断
 -> OTA活动建议
 -> 前台执行清单
 -> 收益最大化方案

竞对预警
 -> 调价建议的参考变量
 -> OTA活动建议的参考变量
 -> 不直接触发自动调价

评论分类与运营反馈
 -> OTA健康诊断
 -> 前台执行清单
 -> 门店整改清单
```

## 4. 公共数据对象

### 4.1 酒店配置

- 酒店类型。
- 房间总数。
- 房型列表。
- 每个房型房间数。
- 每个房型底价。
- 每个房型当前价。
- 每个房型活动后到手价。
- 特殊日期系数。
- 策略模式：满房派、RevPAR派、利润派。
- 固定竞对池。

### 4.2 房型库存

```text
房型可售库存 = 房型总房数 - 已售房数 - 连住房数 - 保留房 - 维修房 - 线下预留房 + 预计退房可售房
房型剩余可售比例 = 房型可售库存 / 房型总房数
```

### 4.3 OTA漏斗

- 曝光量。
- 浏览量。
- 提交订单量。
- 支付订单量。
- 取消订单量。
- 成交间夜。
- 流失量。
- 点评分。
- 点评量。
- 差评量。

### 4.4 历史基准

至少保留以下口径：

- 近7天均值。
- 近14天均值。
- 近30天均值。
- 历史同星期均值。
- 历史同日期属性均值。
- 节假日历史均值。
- 特殊活动历史均值。

## 5. 公共输出分级

### 5.1 OTA健康五档

- 5分：优秀。
- 4分：健康。
- 3分：观察。
- 2分：预警。
- 1分：严重异常。

### 5.2 需求指数四档

- 0-25：低谷。
- 26-50：平淡。
- 51-75：旺日。
- 76-100：爆发。

### 5.3 建议动作强度

- 0：不建议动作。
- 1：观察。
- 2：轻微调整。
- 3：明确调整。
- 4：强干预。

## 6. 第一版优先级

第一优先级：

- 需求指数。
- 调价建议。
- OTA健康诊断。
- 流量转化诊断。
- 竞对预警。

第二优先级：

- OTA活动建议。
- 前台执行清单。
- 收益最大化方案。

第三优先级：

- 评论分类与运营反馈。

## 8. 数据列表校验结论

已根据 `数据列表.txt` 对功能蓝图做第一轮校验。整体结论：

- 可用数据足以支撑第一版算法，不需要额外假设“完全拿不到市场数据”。
- 需求指数可直接使用节假日信息、区域活动、行业热度监控、平台流量峰谷时段、历史同期和订单进度。
- OTA健康诊断应优先接入 HOS 总分、商家经营分、同行竞争力评分、同行均值和同行排名，而不是只依赖自行计算的漏斗指标。
- 流量转化诊断可使用曝光、浏览人数、支付转化率、搜索转化率、一转/二转数据。
- 调价建议可使用每日挂牌价、每日促销价、各房型实时房价、各平台预付价对比、原价、优惠价、优惠详情和竞品优惠价。
- 评论模块可使用评分、差评率、新增评价数、评价内容、客人印象关键词、店家回复内容、差评跟进状态和异常消息。
- 前台执行清单可接入实时房态、今日预抵、今日预离、房态类型、执行任务、操作员和操作时间。
