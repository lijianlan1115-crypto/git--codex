# 数据字段映射与校验记录

版本：v0.1 
日期：2026-05-15

## 1. 校验目标

根据 `数据列表.txt` 和课程逐字稿，对 9 个功能算法进行第一轮可用性校验。

校验重点：

- 现有数据是否能支撑算法计算。
- 算法是否存在缺失字段。
- 是否有更高质量的数据字段可以替代原先的推导字段。
- 是否符合逐字稿中的运营逻辑。

## 2. 总体结论

数据列表可以支撑第一版算法。

主要优势：

- 经营数据完整，包含间夜、销售额、均价、成交金额、核销金额、ADR、RevPAR。
- OTA数据较完整，包含曝光、浏览、支付转化率、一转/二转、流失订单、同行排名。
- 价格数据较完整，包含挂牌价、促销价、原价、优惠价、各平台预付价、竞品优惠价。
- 口碑数据较完整，包含综合评分、维度评分、差评率、评论内容、关键词、回复内容、差评跟进状态。
- 任务数据具备闭环字段，包含执行任务、操作人、操作时间、好评完成量、推广任务进度。

主要缺口：

- 固定成本和单房变动成本未在数据列表中明确出现，利润派需要用户配置。
- 调价频率和单次调价幅度需要继续从逐字稿核对或设计为可配置参数。
- 用户定义的底价需要由系统配置，数据列表中没有直接出现“OTA活动叠加后到手最低价”字段。
- 评论补偿金额和补偿标准需要业务配置。

## 3. 功能字段映射

### 3.1 需求指数与流量峰谷

可用字段：

- 节假日信息。
- 区域活动、行业热度监控。
- 流量峰谷时段。
- 平台曝光量。
- 浏览人数。
- 搜索转化率。
- 支付转化率。
- 未来房间预订数据。
- 对比上年同期、较上周同期、目标达成度。

结论：

算法可运行，应从纯历史模型升级为外部环境和本店进度混合模型。

### 3.2 调价建议

可用字段：

- 当前房价、每日挂牌价、每日促销价。
- 各房型实时房价。
- 原价、优惠价、优惠详情。
- 各平台预付价对比。
- 房价类型、价格模式、自定义价格审批。
- 同区域/同档次酒店优惠价。
- 未来房间预订数据。

结论：

算法可运行。必须增加价格一致性、活动叠加、可售状态和审批状态校验。

### 3.3 OTA健康诊断

可用字段：

- HOS总分及构成。
- 商家经营分。
- 同行竞争力评分。
- 同行均值。
- 同行排名。
- 曝光、浏览、支付转化率。
- 流失订单、流失间夜、流失金额。
- 评分、差评率、评价数。

结论：

算法可运行。建议用平台官方评分加自建漏斗评分，避免重复造一个与平台权重脱节的健康分。

### 3.4 流量转化诊断

可用字段：

- 曝光量。
- 浏览人数。
- 点击量。
- 搜索转化率。
- 一转/二转数据。
- 支付转化率。
- 流失订单数、流失间夜、流失金额。

结论：

算法可运行。优先使用平台一转/二转口径，再使用自算漏斗。

### 3.5 OTA活动建议

可用字段：

- 每日促销价。
- 优惠价。
- 优惠详情/标签。
- 市场营销标签。
- 竞品活动策略。
- 优惠券发放记录。
- 推广消耗金额、推广余额。
- 会员级别。

结论：

算法可运行。活动需要区分价格型、标签型、流量型和会员型。

### 3.6 竞对预警

可用字段：

- 同区域/同档次酒店优惠价。
- 竞品销量排名。
- 竞品活动策略。
- 同行竞争力评分。
- 同行均值。
- 同行排名。
- 商圈份额。

结论：

算法可运行。竞对价格仍是基础字段，销量、评分、活动和份额作为增强解释变量。

### 3.7 前台执行清单

可用字段：

- 实时在住。
- 今日预抵。
- 今日预离。
- 房态类型。
- 各房型实时房价。
- 电脑状态数量。
- 差评跟进状态。
- 执行任务。
- 操作员、操作时间。
- 扫码完成量、好评完成量、推广任务进度。

结论：

算法可运行。应从任务生成升级为任务闭环管理。

### 3.8 收益最大化方案

可用字段：

- 入住间夜。
- 满房率。
- 销售间夜。
- 均价。
- 销售额、成交金额、核销金额。
- 预计出租率、预计ADR、预计RevPAR。
- 结算金额、手续费、退款明细。
- 客源类型、渠道来源。

结论：

满房派和 RevPAR派可直接运行。利润派需要补充固定成本和单房变动成本。

### 3.9 评论分类与运营反馈

可用字段：

- 综合评分。
- 各维度评分。
- 差评率。
- 新增评价数。
- 评论内容。
- 客人印象关键词。
- 店家回复内容。
- 差评跟进状态。
- 好评完成量。

结论：

算法可运行。第一版优先做分类、升级、问题回流，不优先做回复文案风格。

## 4. 逐字稿校验摘要

已核对的关键课程规则：

- 第1条：订单下降按市场、平台、自身运营三层排查。
- 第1条：旺季收尾、节假日前后、暑期结束和开学季会造成需求变化。
- 第1条：平台补贴、平台自动改价、跨平台价差会影响支付转化。
- 第1条：未来3-14天甚至21天某类房型大量关房会影响订单量和排名。
- 第2条：先有量再有价，白天跑量，晚上酌情提价。
- 第2条：18点前冲刺60%+出租率是满房关键观察点。
- 第4条：竞对低价竞争不能盲目跟降，要先识别对方体量、成本和运营背景。
- 第5条：先做转化，再做流量。
- 第8条：动态定价看市场需求、竞对价格、自身预订增速和库存。
- 第15条：稳定满房需要45-60天价格体系相对稳定。
- 第17条：前台每2小时通报库存、后台数据和外网检查。
- 第21条：收益高手看出租率、平均房价、单房收益和毛利。

## 5. 下一步建议

优先深化顺序：

1. 调价建议算法。
2. 需求指数算法。
3. OTA健康诊断算法。
4. 流量转化诊断算法。
5. OTA活动建议算法。

原因：

- 调价建议是用户最关注的直接产出。
- 需求指数是其他功能的底层变量。
- OTA健康诊断决定是否应该先修转化再调价。
- 流量转化诊断决定降价是不是最后手段。
- OTA活动建议与调价高度相关，必须防止活动叠加突破底价。
