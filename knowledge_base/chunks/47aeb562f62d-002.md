## 5. AI 不得声称已经修改或回滚，除非有证据

AI 不得说：

```text
已修复
已回滚
已提交
已推送
```

除非它真的执行了操作，并提供：

```text
git diff
git status
测试结果
修改文件列表
```

如果只是建议，必须说：

```text
建议这样修
可以执行以下命令检查
```

---

## 6. AI 必须保护公共契约

AI 不得随意改：

```text
contracts/v27/contract.json
field_registry
skill_io
node_io
edge_payload
runtime/data_gate.py
runtime/safety/*
```

如果必须改，必须说明：

```text
新增字段是什么
为什么蓝图需要
哪些 Skill 使用
缺失时怎么处理
是否影响 approval/live
```

---

## 7. AI 必须维护上下游边界

AI 不得让：

```text
S5 重算 S16
S6 重算 S5
S11 重算 S8/S10
S14 自己重复造 S9 漏斗
A4 做 A3 的决策
```

AI 应使用：

```text
upstream_result_ref
snapshot_ref
run_context_id
decision_id
s5_output_hash
```

保证交接可追踪。

---

## 8. AI 必须补测试

AI 改完代码必须补对应测试。
没有测试的修改不能算完成。

最低测试：

```bash
python -m compileall runtime
env -u HOTEL_OTA_DEMO_MODE python -m unittest discover tests
python tools/validate_v27_contract.py
python tools/audit_v27_contract_drift.py
```

# 四、人和 AI 共同遵守的交接格式

每个任务开始前必须写：

```text
任务名称：
负责 Agent：
负责 Skill：
负责节点：
上游输入：
下游输出：
涉及字段：
是否新增字段：
是否影响 contract：
是否影响 approval/live：
是否需要测试：
```

示例：

```text
任务名称：深化 S5 调价建议算法
负责 Agent：A3 Revenue Strategist
负责 Skill：S5
负责节点：N015
上游输入：S2 snapshot、S4 demand_index、S9 conversion_diagnosis、S16 progress_deviation
下游输出：S6 pricing_execution_dry_run
涉及字段：expected_occupancy_tonight、room_type_base_price、price_floor、price_ceiling
是否新增字段：是
是否影响 contract：是
是否影响 approval/live：是，缺关键字段时阻断 S6
是否需要测试：是
```

# 五、合并前检查清单

合并前必须确认：

```text
1. 是否按蓝图实现，而不是按 sample 数据硬编码？
2. 是否新增字段？如果新增，是否写入 field_registry？
3. 是否更新 skill_io / node_io / edge_payload？
4. 是否有 data_gap 策略？
5. 是否标记 demo_data / sample_data / synthetic_today_demo？
6. 是否影响 approval_data_allowed / live_allowed？
7. 是否破坏上游或下游？
8. 是否有测试？
9. 是否通过 compileall、unittest、contract 校验？
10. git status 是否干净？
```

# 六、最重要的一句话

本项目协作的核心不是“谁改了哪些代码”，而是：

```text
谁负责哪个 Agent / Skill 的算法边界；
谁维护哪个输入输出契约；
谁保证缺数据时不乱算；
谁保证下游不会拿错误数据继续执行。
```

代码只是实现，契约和边界才是协作的核心。
