# OTA活动建议算法

## 1. 功能名称

OTA活动建议。

## 2. 目标

判断当前是否应该参加、保留、取消或调整 OTA 活动，并说明活动对价格、转化率、客群和平台间价格一致性的影响。

第一版只输出建议，不自动报名或取消活动。

## 3. 输入数据

- 当前活动列表。
- 活动类型：天天特价、新客、早定、连住、限时特惠、今夜特价、红包补贴、场景活动。
- 每个活动折扣。
- 活动叠加规则。
- 活动后到手价。
- 房型底价。
- 新客订单占比。
- 老客订单占比。
- 早定订单占比。
- 连住订单占比。
- 曝光量变化。
- 浏览量变化。
- 支付转化率变化。
- 取消率变化。
- 平台间价格差。
- 竞对活动预警。
- 需求指数。
- 房型剩余可售比例。

## 4. 核心指标

```text
活动后到手价 = 原价 × 活动折扣 × 其他叠加折扣 - 平台补贴影响
底价安全差 = 活动后到手价 - OTA活动叠加后底价
活动转化提升率 = 活动后转化率 / 活动前转化率 - 1
活动订单提升率 = 活动后订单量 / 活动前订单量 - 1
平台价差率 = abs(平台A到手价 - 平台B到手价) / min(平台A到手价, 平台B到手价)
```

## 5. 活动价格秩序

课程线索之一：

```text
早定/连住 ≤ 今夜特价 = 限时特惠 ≤ 新客 ≤ 天天特价
```

课程线索之二：

```text
早定/连住 ≤ 天天特价 ≤ 新客折扣
当天限时特惠/今夜特价 ≤ 新客折扣
```

第一版处理方式：

- 把活动价格秩序做成可配置模板。
- 默认模板采用“早定/连住最低，限时/今夜用于临时刺激，新客和老客差异不要太大”。
- 不把单一排序写死，后续需继续核对逐字稿。

## 6. 判断逻辑

### 6.1 是否需要上活动

建议上活动的条件：

```text
需求指数低或平淡
AND 库存高
AND 转化率不低到需要先整改
AND 活动后到手价 >= 底价
```

不建议上活动的条件：

```text
活动后到手价低于底价
OR 当前转化率异常低且未完成诊断
OR 平台间价格差已经异常
OR 需求爆发且库存低
```

### 6.2 是否保留活动

保留条件：

```text
活动后转化率提升
AND 活动后订单提升
AND 未突破底价
AND 未造成平台间价格异常
```

### 6.3 是否取消活动

取消条件：

```text
活动后浏览支付转化率下降
OR 平台间价格差扩大
OR 低价订单占比过高
OR 需求旺且库存不足
OR 活动后到手价接近或低于底价
```

### 6.4 限时特惠/今夜特价

适用场景：

- 当天订单少。
- 降价需要直接刺激支付。
- 淡季新客定向降价效果差。
- 需要标签露出提高推流权重。

风险：

- 早定高价客人可能取消重订。
- 平台规则不同，部分平台不能只对新客露出。
- 可能引发差评或价格争议。

## 7. 红包补贴判断

课程线索：

- 红包补贴不是面向所有客人。
- 会加剧千人千面价格呈现。
- 多平台运营时，参加甲平台补贴可能影响乙平台支付转化。
- 单平台运营可积极参与。
- 多平台运营需看转化率数据决定保留或取消。

判断：

```text
如果单平台为主 -> 可积极参与补贴
如果多平台运营 -> 观察参与后浏览转化和支付转化
如果转化率下降 -> 建议取消或协调平台
如果转化率正常或提升 -> 建议保留
```

## 9. 可配置参数

- 活动价格秩序模板。
- 活动后底价安全距离。
- 活动观察周期。
- 平台价差异常阈值。
- 低价订单占比阈值。
- 新客/老客折扣最大差异。
- 早定/连住最低折扣规则。

## 10. 数据列表适配校验

### 10.1 可直接使用的数据

来自 `数据列表.txt` 的可用字段：

- 每日促销价。
- 优惠价。
- 优惠详情/标签。
- 市场营销标签。
- 价格模式。
- 房价类型。
- 各平台预付价对比。
- 竞品活动策略。
- 充值活动配置。
- 优惠券发放记录。
- 推广消耗金额。
- 推广余额。
- 会员级别。
- 销量、订单来源明细、渠道名称。
- 曝光量、浏览人数、支付转化率。
- 流失订单数、流失间夜、流失金额。

### 10.2 算法修正

OTA活动建议必须先判断“活动是价格工具、流量工具，还是标签工具”。

活动类型分层：

```text
价格型活动：直接改变到手价，例如促销价、优惠券、折扣。
标签型活动：主要为了露出标签，例如今夜特价、限时特惠。
流量型活动：主要获得平台资源位或推广曝光。
会员型活动：影响老客、新客和会员价差。
```

修正后的判断顺序：

```text
1. 活动后到手价是否低于底价？
2. 活动是否造成平台间价格异常？
3. 活动目标是拉曝光、拉浏览、拉支付，还是清库存？
4. 活动上线后是否提升对应指标？
5. 活动是否造成低价订单占比过高？
6. 是否影响未来几天价格体系稳定？
```

### 10.3 逐字稿校验依据

逐字稿第9条确认：活动折扣要利于全面获客，不能只顾新客忽略老客，折扣差异不能过大。

逐字稿第10条确认：先建秩序、锁逻辑、定原价，再动态调价，日常调价尽量只改原价不乱改折扣。

逐字稿第11条确认：红包补贴会影响平台间转化，多平台运营时要观察转化率决定保留或取消。

### 10.4 准确性与效率结论

活动算法应避免“看到订单少就上活动”。更准确的做法是先判断漏斗瓶颈，再选择价格型、标签型、流量型或会员型活动。
