# 酒店 OTA AI 数字员工项目协作原则

## 一、总原则

本项目不是普通功能堆砌项目，而是一个基于 **功能蓝图 → Skill 算法 → 字段契约 → 节点链路 → 多 Agent 协作 → 飞书端输出** 的系统。

所有人和 AI 都必须遵守：

1. **先看蓝图，再写代码。**
2. **先确定 Skill 输入输出，再实现逻辑。**
3. **先对齐字段契约，再新增字段。**
4. **先说明准备改什么，再修改文件。**
5. **缺数据必须明确 data_gap，不允许用 sample 值冒充真实算法。**
6. **demo_data / sample_data / synthetic_today_demo 必须明确标记，不能说成真实经营数据。**
7. **A3 只做建议，A4 才做执行边界；任何模块都不能绕过审批和 live 闸门。**
8. **公共契约、公共 runtime、数据闸门、安全模块不能随意改。**

# 二、给人类开发者的协作原则

## 1. 按 Agent / Skill 分工，不按文件随便分工

开发者必须清楚自己负责哪个 Agent、哪个 Skill、哪个节点。

建议分工：

| Agent | 职责 | 节点 | 边界 |
| --------------------- | ------- | -------------- | ------------------------------ |
| A2 Diagnosis Analyst | 诊断与目标 | N009/N010/N011 | 只做诊断、目标、偏差归因，不做调价执行 |
| A3 Revenue Strategist | 收益与推广决策 | N012/N013/N015 | 只输出策略建议、dry-run/审批建议，不 live 写入 |
| A4 Execution Guard | 执行与审批 | N016/N017/N021 | 只做执行校验、审批、回读、安全闸门，不重新决策 |

禁止：

* A3 直接写 OTA / PMS。
* A4 重新计算调价策略。
* S5 自己重算 S16 的进度偏差。
* S6 自己决定价格。
* 任意 Skill 自己临时造一套数据源。

---

## 2. Contract First：先契约，后代码

开发前必须先看：

```text
contracts/v27/contract.json
contracts/v27/field_registry.*
contracts/v27/skill_io_contract.*
contracts/v27/node_io_contract.*
contracts/v27/edge_payload_contract.*
```

新增字段必须写清：

```text
canonical_key: 英文字段名
zh_name: 中文含义
source_blueprint: 来源蓝图
source_system_candidate: PMS / OTA / 外部 / 算法输出 / 人工配置
used_by_skill: 用在哪些 Skill
produced_by_node: 哪个节点产出
consumed_by_node: 哪个节点消费
missing_policy: 缺失时怎么办
fallback_policy: 可否兜底
approval_impact: 是否影响审批
live_impact: 是否影响 live
```

禁止出现：

```text
今晚预期出租率
expectedOcc
expected_occupancy
expectedOccupancy
```

统一使用：

```text
expected_occupancy_tonight
```

以下文件属于保护区，不能随便改：

如果必须改，要单独说明：

```text

影响哪些字段
是否破坏旧测试
如何回滚
```

## 4. 上下游只能通过固定 payload 交接

正确方式：

```text
S16 输出 progress_deviation_result
S5 读取 progress_deviation_result

S5 输出 pricing_decision
S6 读取 pricing_decision

S8/S10 输出 promotion_plan / roi_decision
S11 读取 promotion_plan / roi_decision
```

错误方式：

```text
S5 自己重新算进度偏差
S6 自己重新算建议价
S11 自己重新决定投放策略
```

---

## 5. 缺数据必须返回 data_gap

缺这些字段时，不允许硬编码：

```text
expected_occupancy_tonight
room_type_base_price
room_type_sellable_inventory
submitted_orders
activity_final_price
promotion_roi
competitor_price_index
price_floor
price_ceiling
freshness_status
```

必须返回类似结构：

```json
{
 "status": "data_gap",
 "missing_fields": ["expected_occupancy_tonight"],
 "affected_skills": ["S5"],
 "blocked_downstream": ["S6"],
 "confidence_score": 0.35,
 "can_generate_preview": true,
 "can_handoff_to_execution": false,
 "approval_data_allowed": false,
 "live_allowed": false
}
```

---

## 6. demo/sample/synthetic 数据不能混用

必须明确数据类型：

| 数据类型 | 含义 | 能否审批 | 能否 live |
| -------------------- | --------------- | ---: | --------: |
| sample_data | 兜底样例 | 否 | 否 |
| demo_data | 固定演示数据 | 否 | 否 |
| synthetic_today_demo | 当天模拟演示数据 | 否 | 否 |
| database_fresh | 真实/试运行 fresh 数据 | 视权限 | 视 live 开关 |
| database_historical | 历史数据 | 否 | 否 |

同一条链路必须共享：

```text
run_context_id
hotel_id
business_date
data_source_type
freshness_status
snapshot_ref
```

不允许出现：

```text
S2 用 demo_data
S4 用 sample_data
S14 用另一套 sample
S5 自己造数据
```

## 7. 每个 PR 必须带测试

最低要求：

```bash
python -m compileall runtime
env -u HOTEL_OTA_DEMO_MODE python -m unittest discover tests
python tools/validate_v27_contract.py
python tools/audit_v27_contract_drift.py
```

每个 Skill 要有对应测试：

```text
tests/skills/test_s14_ota_health.py
tests/skills/test_s15_sales_baseline.py
tests/skills/test_s16_progress_deviation.py
tests/skills/test_s5_pricing_decision.py
tests/skills/test_s6_price_dry_run.py
```

---

## 8. PR 要小，不要一口气乱改

推荐 PR 粒度：

```text
feat(a2): implement S16 progress deviation with data gap policy
feat(a3): deepen S5 pricing decision tree from blueprint
fix(a4): block S6 dry-run when price floor is missing
test(skill): add S9 conversion funnel missing submitted_orders test
```

不推荐：

```text
fix everything
update all files
big refactor
```

# 三、给 AI / Codex 的协作原则

## 1. 修改前必须先说明准备改什么

AI 每次改代码前必须输出：

```text
准备改什么：
1. 修改哪些文件
2. 为什么要改
3. 依据哪个蓝图 / contract / 测试
4. 可能影响哪些 Skill
5. 如何验证
6. 如何回滚
```

不允许直接上来就改文件。

---

## 2. AI 必须先读项目事实源

AI 开始任务前必须先读：

```text
README.md
contracts/v27/contract.json
contracts/v27/*
runtime/*
tests/*
docs/*
功能逻辑蓝图
```

优先级：

```text
功能蓝图算法 > V27 contract > 当前 runtime > demo 数据 > legacy 文档
```

禁止只看当前代码就下结论。

---

## 3. AI 不能为了让测试通过而硬编码

禁止：

```text
demand_index = 58
ota_health_score = 4
promotion_roi = "observe"
contains_real_customer_data=true 但 replay_status=passed
```

如果缺数据，必须返回 data_gap。
如果是 demo/sample，必须标明低置信度。

---

AI 不得默认修改：

```text
/etc/hotel-ota-ai/*
.env
runtime/cli.py 自动加载 /etc env
systemd service
Git remote
Git branch
```
