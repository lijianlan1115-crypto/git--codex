# S16 进度偏差诊断规则

## 算法来源
- 对应节点：N011 / S16 进度偏差诊断。
- 对应 Agent：A2。
- 对应 BP：P0。
- 源文件索引：`references/source/source_manifest.yaml`。
- 字段契约：`contracts/node_io_contract.yaml`。
- runtime 规则：`runtime/algorithm_rules/deviation_rules.yaml`。
- 阈值、节点完成率分级和下游路由开关必须以 runtime 规则文件为准；skill 只负责加载、解释和输出，不自行发明阈值。
- 如果上述 source、contract 或 runtime 规则文件缺失，必须输出 `data_gap` 或 `config_gap`，不得把缺失配置伪装成 confirmed 规则。

## 输入分层
- hard_required：`hotel_id`、`data_business_date`。缺失时必须阻断主链路，返回 `missing_fields`，`confidence=low`。
- soft_required：`actual_orders`、`target_orders`、`progress_checkpoint`、`checkpoint_target_orders`、`field_freshness`。缺失时可以继续输出 `data_gap` 或历史复盘，但不得触发 S5。
- optional：`exposure`、`views`、`clicks`、`paid_orders`、`payment_conversion_rate`、`demand_index`、`occupancy_rate`。
- candidate：`sold_rooms`、`occupied_rooms` 只能作为实际进度代理证据，不得作为 live 调价依据。
- blocked_for_live：`demo_data`、`sample_data`、`stale`、`missing_date`。这些口径只能用于诊断、预览或 dry-run，不得创建正式审批或进入 live 执行。

## 判断逻辑
1. 优先执行 DataGate：检查 hard_required、业务日期、字段质量和数据新鲜度。
2. 优先采用 runtime `deviation` 输出，不在 skill 中重写确定性计算。
3. 按 12 点、16 点、20 点节点目标判断当前进度，不能只看全天目标；若上游提供 23 点收口节点，只作日终复盘或全天收口说明。
4. `actual_orders` 优先使用 `order_snapshot.business_date` 或 `order_snapshot.checkin_time` 的日期部分统计今日唯一订单数。
5. `actual_orders` 不得取 `daily_metrics.room_nights`；缺失今日实时订单/今日已售字段时只能输出 `data_gap`。
6. `target_orders` 来自 S15 基准线或明确日目标；必须输出 `actual_source`、`target_source`、`actual_basis_date`、`target_basis_date` 和 `field_freshness`。
7. 节点完成率 `checkpoint_completion_rate` 是主判断口径；`daily_completion_rate` 只作辅助。
8. 当 `checkpoint_completion_rate >= 0.9` 时，方向可解释为 `ahead`。
9. 当 `0.65 <= checkpoint_completion_rate < 0.9` 时，方向可解释为 `normal`。
10. 当 `checkpoint_completion_rate < 0.65` 时，方向可解释为 `behind`。
11. 节点进度落后时先区分流量不足和转化不足。
12. 只有流量不足时，优先建议推广、曝光入口、活动资源和内容优化，不直接降价。
13. 只有转化不足且 `traffic_problem=false`、`downstream_allowed=true` 时，才允许进入 S5 调价或活动调整候选。
14. 任一核心字段不是可信今日口径时，不得触发 S5，只能转 S14 或提示数据缺口。
15. 输出路由给 S14/S9/S5/S3 时，必须保留 `freshness_status`、`data_business_date`、`field_freshness`、`downstream_allowed` 和 `downstream_blocked_reason`。

## 阈值和冲突处理
- 阈值与分级必须读取 `runtime/algorithm_rules/deviation_rules.yaml`。
- 当前 runtime 分时节点为 `checkpoint_hours=[12,16,20]`。
- 当前 runtime 完成率阈值为：`ahead >= 0.9`，`normal >= 0.65`，`behind < 0.65`。
- S5 候选必须同时满足：实际订单 fresh、目标可信、方向为 `behind`、转化不足、非流量不足。
- 冲突处理顺序：DataGate > freshness > approval/live guard > price/budget guard > skill-specific threshold。

## 房态和销量口径
- 入住销量按入住当夜统计。
- 次日下午 14:00 前退房且未续住的房间，按当日可售房处理。
- 数据上传时间不能替代业务时间；必须区分 `data_snapshot_time` 和订单/入住业务日期。

## 数据库来源
- 可读取 `database-query --template operating_snapshot`、`daily_metrics`、`monthly_metrics` 和 `order_snapshot`。
- 预订明细应先归一化为 `order_snapshot` 或 `reservation_snapshot`；不要在 skill 里写死 JD01/JD04/RS01 表名。
- MySQL 报表库只作为偏差诊断证据来源，目标口径不一致时输出 `needs_business_confirm`。
- 月/年累计只能作为对比证据，不能作为今日节点目标。
- 旧数据只能输出历史/演示偏差，不得触发 S5 正式调价审批。

## 输出要求
- confirmed outputs：`direction`、`progress_checkpoint`、`checkpoint_target_orders`、`checkpoint_completion_rate`、`actual_orders`、`target_orders`。
- candidate outputs：`traffic_problem`、`conversion_problem`、`pricing_candidate_allowed`、`downstream_skill`。
- 必须输出 `actual_source`、`target_source`、`actual_basis_date`、`target_basis_date` 和 `field_freshness`。
- 必须输出 `downstream_allowed` 和 `downstream_blocked_reason`，说明是否允许继续进入 S5/S6 调价链路。
- 必须输出 `freshness_status`、`data_business_date`、`data_snapshot_time`、`business_status` 和 `today_label_allowed`。
- 飞书回复只输出脱敏摘要，不贴完整 runtime JSON、数据库原始表、配置或源码。

## 降级规则
- hard_required 缺失时返回 `missing_fields`，`confidence=low`，并阻断主链路。
- soft_required 缺失时继续生成 `data_gap` 或历史复盘，但必须输出 `data_gap` 和降级原因。
- 输入为 `demo_data`、`sample_data`、`stale` 或 `missing_date` 时，只能返回 `preview_only` 或 `dry_run`，不得创建正式审批或进入 live。
- 实际订单不是同日期 fresh 实时口径时，必须设置 `direction=data_gap` 或等价阻断状态，并将下游限制到 S14。
- 目标订单不可信时，必须输出 `downstream_blocked_reason=target_orders_not_trusted` 或等价说明。
- 来源能力为 `read_only` 或 `manual` 时，只能输出建议、诊断或人工任务，不执行写动作。

## 安全规则
- 真实调价、房量、推广、评论发布必须审批。
- S16 不直接执行写动作，只能给 S14/S9/S5 提供诊断信号。
- 输出必须带 `data_business_date`、`data_snapshot_time`、`freshness_status`。
- `sample_data`、`demo_data`、`stale`、`missing_date` 不能创建正式调价审批。
