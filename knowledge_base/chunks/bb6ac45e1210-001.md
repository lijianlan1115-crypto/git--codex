# 运营安全策略

## 角色权限

- `admin` / 超级管理员：可管理角色表、安全配置、所有 skill、所有审批和紧急停用。
- `owner` / 老板：可查看全部诊断和收益建议，可审批真实执行动作，可调整业务安全阈值。
- `operator` / 运营：可触发诊断、生成建议、准备 dry-run 动作、发起审批、跟踪结果。
- `frontdesk` / 前台：可接收任务、上传截图、反馈完成情况；不得直接触发调价、推广或口碑发布。
- `guest` / 未授权用户：飞书中只返回无权限提示，不读取业务数据，不触发业务 skill。

飞书触发必须同时满足：

1. 群聊在 `groupPolicy=allowlist` 范围内。
2. 群聊消息明确 @ 机器人。
3. 发送人能从角色表解析为 `admin`、`owner`、`operator` 或 `frontdesk`。
4. 当前动作在该角色权限内。

拿不到飞书发送人身份时，所有业务动作按 `guest` 阻断；只允许返回“请联系管理员加入角色表”的提示。

## 飞书生产边界

- 飞书生产入口禁止导出配置包、env、auth profile、角色表、数据库映射、OpenClaw 主配置、源码 zip、CSV、XLSX、JSON 原始数据和完整 runtime JSON。
- 飞书生产入口禁止展示代码片段、真实 `open_id/chat_id`、DSN、API 请求体和字段映射全量内容。
- 飞书业务 Agent 不得创建、修改、删除项目文件、角色表、数据库映射、环境变量或源码。
- 需要记录问题时，只输出问题记录模板内容，由开发者在本地开发流程提交。

## 执行闸门

1. 先校验 `auth_context` 和角色权限。
2. 先解释建议。
3. 展示证据和风险。
4. 生成 dry-run 动作。
5. 调价、房量、房态、推广、公开回复都必须 `admin` 或 `owner` 审批。
6. 审批通过、安全校验通过、live 开关开启后才允许调用真实接口。
7. 记录请求摘要、响应码、失败原因和 fallback。

正式审批必须先生成 `approval_id`，并包含数据新鲜度和 dry-run 摘要。旧数据、sample/demo 数据不得创建正式审批。

## Fallback 顺序

1. Beyondh API。
2. RPA 或浏览器自动化。
3. 飞书下发前台人工任务。

## 参考资料边界

Xhotel/小满、超级店长、甩手掌柜只能帮助理解业务场景：

- 7x24 监控。
- 诊断和运营指令。
- 调价、流量、推广、口碑、任务闭环。

它们不是本项目验收标准。验收标准以 OpenClaw skill 工程包、需求文档、实施教程和验收用例为准。
