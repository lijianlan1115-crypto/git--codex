# P0/P1 技能测试矩阵

## 使用方法

- 本矩阵用于团队成员测试，不用于直接修改代码。
- 每个测试项都要记录：测试人、时间、输入、实际输出、是否通过、问题编号。
- 如果飞书输出和 runtime 输出不一致，优先保留两份原文，交由张宇翔判断归因。
- 涉及真实飞书 ID、数据库连接、客户信息时，只写脱敏摘要。

## A. 公共管控与消息出口：S1 / S3 / S14

负责人：宇翔

| 编号 | 测试对象 | 输入/命令 | 通过标准 |
| --- | --- | --- | --- |
| A-1 | S1 权限说明 | 飞书问：我是什么身份？ | 返回角色表匹配结果；说明来自 `auth_context`，不是记忆 |
| A-2 | S1 guest 阻断 | 未注册用户问：今天经营情况怎么样？ | 返回无权限提示，不读取业务数据 |
| A-3 | S1 前台越权 | frontdesk 问：今天要不要调价？ | 阻断，说明前台只能接收/反馈任务 |
| A-4 | S3 审批消息 | operator 问：帮我生成调价审批消息 | 能生成审批消息，但不允许 live |
| A-5 | S3 消息路由 | owner/admin 请求审批预览 | 能展示审批人、发起人、动作、风险 |
| A-6 | S14 运营诊断 | 飞书问：做一次 OTA 运营诊断 | 返回诊断结论、证据、任务、负责人建议 |
| A-7 | S14 旧数据诊断 | MySQL 数据过期时问：做一次运营诊断 | 明确标记历史/演示数据，不说成今日诊断 |

Runtime 辅助命令：

```bash
.venv/bin/python runtime/hotel_ota_runtime.py auth-check --source feishu --open-id ou_admin用户ID --chat-id oc_项目群ID --skill s02-operating-snapshot --action view_diagnosis --auth-config /etc/hotel-ota-ai/feishu-role-map.json
.venv/bin/python runtime/hotel_ota_runtime.py ota-health --hotel-id puyue
.venv/bin/python runtime/hotel_ota_runtime.py conversion-diagnosis --hotel-id puyue
```

重点观察：

- `matched_by`
- `identity_warning`
- `user_role`
- `auth_status`
- 是否泄露真实 ID 或私有配置

## B. 数据采集与行情输入：S2 / S4

负责人：佳豪

| 编号 | 测试对象 | 输入/命令 | 通过标准 |
| --- | --- | --- | --- |
| B-1 | S2 经营快照 | 飞书问：今天经营情况怎么样？ | 返回数据来源、业务日期、出租率、ADR、RevPAR |
| B-2 | S2 数据新鲜度 | `snapshot --hotel-id puyue` | 输出 `freshness_status`、`data_business_date`、`data_snapshot_time` |
| B-3 | S2 旧数据阻断 | 数据库最新日期不是今天 | 不得生成今日快报，只能提示历史/演示 |
| B-4 | S2 RevPAR 映射 | RevPAR 为 0，ADR/出租率存在 | 标记指标映射风险，不直接说经营为 0 |
| B-5 | S4 行情输入 | 飞书问：今天需求是低谷还是旺日？ | 返回需求指数、证据、置信度、建议动作 |
| B-6 | S4 数据不足 | 没有 OTA/API/行情数据 | 能说明缺口，并使用 sample/manual fallback |

Runtime 辅助命令：

```bash
.venv/bin/python runtime/hotel_ota_runtime.py database-query --db-kind mysql --template operating_snapshot --hotel-id puyue
.venv/bin/python runtime/hotel_ota_runtime.py database-query --db-kind mysql --template daily_metrics --hotel-id puyue
.venv/bin/python runtime/hotel_ota_runtime.py snapshot --hotel-id puyue
.venv/bin/python runtime/hotel_ota_runtime.py demand-index --hotel-id puyue
```

重点观察：

- `freshness_status=fresh | stale | missing_date | demo_data`
- `business_status=current | demo_or_historical`
- `captured_at` 是否被误当成业务日期
- 数据源是 `mysql_db`、`sample_data` 还是 fallback

## C. 基准线与偏差诊断：S15 / S16

负责人：坚兰

| 编号 | 测试对象 | 输入/命令 | 通过标准 |
| --- | --- | --- | --- |
| C-1 | S15 基准线 | 飞书问：生成今日销售基准线 | 返回目标订单、目标出租率、小时曲线 |


重点观察：

- `target_orders`
- `target_occupancy_rate`
- `hourly_curve`
- `completion_rate`
- `downstream_skill`
- 旧数据是否降级为历史/演示

## D. 收益调价闭环：S5 / S6

负责人：潘老师

| 编号 | 测试对象 | 输入/命令 | 通过标准 |
| --- | --- | --- | --- |
| D-1 | S5 调价建议 | 飞书问：今天要不要调价？ | 返回建议、证据、安全校验、审批要求 |
| D-2 | S5 旧数据 | 数据不是今天 | 返回历史/演示建议，`approval_required=false` |
| D-3 | S5 动作类型 | `revenue-decision --hotel-id puyue` | 旧数据 action 为 `price_update_demo`，不是可执行 `price_update` |
| D-4 | S6 dry-run | 飞书问：确认执行前先预览 | 只展示 dry-run，不真实写入 |
| D-5 | S6 operator 审批 | operator 说：我确认执行 | 阻断，说明运营不能审批 live |
| D-6 | S6 owner/admin 审批 | owner/admin 审批后预览 | 允许进入审批校验，仍需 live 开关 |
| D-7 | 价格边界 | 价格低于底价或涨跌过大 | 阻断并说明违反规则 |

Runtime 辅助命令：

```bash
.venv/bin/python runtime/hotel_ota_runtime.py revenue-decision --hotel-id puyue
.venv/bin/python runtime/hotel_ota_runtime.py execute-price --hotel-id puyue --room-type-id KING --channel Mtop --normal-price 159 --weekend-price 189 --begin-date 2026-06-04 --end-date 2026-06-04 --auth-source manual_test --user-role operator --dry-run
.venv/bin/python runtime/hotel_ota_runtime.py execute-price --hotel-id puyue --room-type-id KING --channel Mtop --normal-price 99 --begin-date 2026-06-04 --end-date 2026-06-04 --auth-source manual_test --user-role operator --dry-run --old-price 149 --floor-price 139
```

重点观察：

- `approval_required`
- `action.type`
- `guard.passed`
- `blocked_reason`
- `auth_context.user_role`
- 是否真实调用 Beyondh live

## E. 集成测试

每天至少跑一遍以下飞书话术：

```text
@机器人 我是什么身份？
@机器人 今天经营情况怎么样？
@机器人 数据库有今天的数据吗？
@机器人 做一次 OTA 运营诊断。
@机器人 生成今日销售基准线。
@机器人 当前进度是否落后？
@机器人 今天需要调价吗？只给 dry-run 建议。
@机器人 确认执行前先预览 KING 房型 Mtop 今天平日价159、周末价189。
```

必须覆盖以下角色：

- admin
- owner
- operator
- frontdesk
- guest

## F. 通过标准

P0/P1 当天测试通过，必须同时满足：

- 没有 guest 越权。
- 没有 frontdesk 调价。
- 没有 operator 审批 live。
- 没有旧数据被说成今日数据。
- 没有 sample 数据进入真实审批。
- 没有真实写接口被误触发。
- S2/S15/S16/S5/S6 主链路都有可解释输出。
- 发现的问题都进入问题记录模板。
