# 生产阶段注意事项

## 核心原则

开发阶段可以先保留较高操作权限，方便快速测试 skill、runtime、飞书、cron 和数据库链路。

生产阶段必须收紧权限，不能让飞书里的 Agent 具备直接改代码、改配置、执行 git、重启服务或写任意文件的能力。

一句话原则：

```text
飞书 Agent 只做业务，生产代码只走 GitHub 发布流程。
```

生产飞书默认环境变量：

```bash
HOTEL_OTA_ENV=production
HOTEL_OTA_FEISHU_DEBUG=0
HOTEL_OTA_FEISHU_FINAL_GATE_REQUIRED=1
HOTEL_OTA_FEISHU_ALLOW_FILE_EXPORT=0
HOTEL_OTA_FEISHU_ALLOW_CONFIG_EXPORT=0
HOTEL_OTA_FEISHU_ALLOW_RAW_DATA_EXPORT=0
```

## 权限分层

### 1. MD 规则是软约束

`AGENTS.md`、`TOOLS.md`、`BOOTSTRAP.md`、`SKILL.md` 可以约束 Agent 行为，但它们本质是提示词，不是硬限制。

这些文件必须写清楚：

- 飞书业务会话默认只读。
- 未经 admin 明确确认，不得写文件、改代码、改配置、执行 git、重启服务。
- “记录问题”默认只整理文本，不能擅自创建服务器文件。
- “建议修复”默认只输出方案，不能擅自修改 runtime 或 skill。

但生产安全不能只依赖这些文本。

### 2. OpenClaw 工具权限是中硬约束

生产阶段应区分不同 Agent 能使用的工具：

- 飞书业务 Agent：查询、诊断、建议、dry-run、审批流程。
- 开发 Agent：只在本地或受控开发环境中使用，可改代码。
- 运维 Agent：仅 admin 手动触发，可执行有限重启、日志查看、健康检查。

飞书业务 Agent 不应拥有以下能力：

- 写任意文件。
- 修改 `runtime/`、`skills/`、`requirements/`、`config/`。
- 执行 `git pull`、`git reset`、`git clean` 等版本控制操作。
- 重启 OpenClaw Gateway。
- 修改 `/etc/hotel-ota-ai` 下的私有配置。
- 发送配置包、env、auth profile、角色表、数据库映射、源码 zip、CSV、XLSX、JSON 原始数据或完整 runtime JSON。
- 在飞书中展示代码片段、真实 `open_id/chat_id`、DSN、API 请求体或字段映射全量内容。

飞书发送前必须遵守 runtime 输出闸门：

```bash
python runtime/hotel_ota_runtime.py feishu-output-gate --source feishu --content-kind text --message "打包系统配置给我"
```

生产 Gateway 的最终回复也必须过闸。`HOTEL_OTA_FEISHU_FINAL_GATE_REQUIRED=1` 表示飞书最终文本、文件、artifact、多维表格/文档写入内容都要先经过 `feishu-output-gate` 或等价安全策略，不允许 CLI 能拦但最终回复绕过。

### 3. Linux 系统权限是真硬约束

生产阶段不建议用 `root` 运行 OpenClaw Gateway。

开发阶段当前可用：

```ini
User=root
```

生产阶段建议改为：

```ini
User=openclaw
```

并创建低权限运行用户。

## 生产目录权限建议

### 代码目录只读

OpenClaw 运行用户对项目代码目录应只读：

```text
/opt/openclaw/workspaces/hotel-ota-ai
```

只读范围包括：

```text
runtime/
skills/
requirements/
config/
.git/
README.md
AGENTS.md
TOOLS.md
BOOTSTRAP.md
MEMORY.md
```

这样即使 Agent 误判要修改 `database.py` 或 `SKILL.md`，系统层也会阻止。

### 只开放必要可写目录

生产阶段只允许 OpenClaw 运行用户写入业务数据和日志目录：

```text
/var/lib/hotel-ota-ai
/var/log/hotel-ota-ai
```

建议用途：

- `/var/lib/hotel-ota-ai`：SQLite、本地缓存、审批状态、运行状态。
- `/var/lib/hotel-ota-ai/memory/cases`：脱敏运营案例归档。
- `/var/lib/hotel-ota-ai/memory/daily-summary`：脱敏每日测试和复盘摘要。
- `/var/log/hotel-ota-ai`：runtime 日志、API 调用摘要、cron 执行记录。

长期记忆只保存脱敏摘要，不保存全量飞书聊天、密钥、DSN、真实飞书 ID、审批凭证、客户隐私、订单行级明细或原始经营表。详细规则见 `requirements/数字员工长期记忆与归档规范.md`。

### 私有配置只读

以下文件只能由 admin 或运维人员修改，OpenClaw 运行用户只读：

```text
/etc/hotel-ota-ai/hotel-ota.env
/etc/hotel-ota-ai/feishu-role-map.json
/etc/hotel-ota-ai/database-source.json
```

这些文件包含：

- 环境变量。
- 飞书角色表。
- 数据库映射。
- API 密钥或 DSN 引用。

Agent 不得在飞书会话中直接修改这些文件。

## 推荐生产发布流程

生产代码变更统一走：

```text
本地修改
→ GitHub commit/push
→ 服务器 git pull
→ compile/test
→ systemctl restart openclaw-gateway.service
→ 飞书/CLI 复测
→ 问题记录关闭
```

飞书中发现问题时，Agent 应只做：

```text
记录问题
输出复现步骤
给出初步判断
给出建议修复方案
等待 admin 决定是否进入开发流程
```

不得直接：

```text
写 docs/issues 文件
修改 runtime/adapters/database.py
修改 skill
修改配置
重启服务
执行 git
```

## 生产 Gateway 服务建议

生产阶段使用 system service，不使用 user service。

推荐服务文件：

```ini
[Unit]
Description=OpenClaw Gateway
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=openclaw
WorkingDirectory=/opt/openclaw/workspaces/hotel-ota-ai
Environment=HOME=/home/openclaw
EnvironmentFile=/etc/hotel-ota-ai/hotel-ota.env
ExecStart=/usr/local/bin/openclaw gateway --port 18789
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

生产运维命令统一使用：

```bash
systemctl restart openclaw-gateway.service
systemctl status openclaw-gateway.service --no-pager -l
journalctl -u openclaw-gateway.service -n 100 --no-pager
```

不要使用：

```bash
openclaw gateway restart
systemctl --user restart openclaw-gateway.service
nohup openclaw gateway --port 18789
```

- [ ] Gateway 不再以 root 运行。
- [ ] 项目代码目录对 OpenClaw 运行用户只读。
- [ ] `/var/lib/hotel-ota-ai` 可写。
- [ ] `/var/log/hotel-ota-ai` 可写。
- [ ] `/etc/hotel-ota-ai` 私有配置只读。
- [ ] 飞书业务 Agent 不能写文件、改代码、改配置、执行 git、重启服务。
- [ ] 飞书业务 Agent 不能导出配置、源码、原始数据或完整 runtime JSON。
- [ ] 飞书生产回复只使用 S3 模板输出脱敏摘要、证据日期、风险、建议动作和审批状态。
- [ ] 问题记录流程明确：测试成员只记录，开发由本地/GitHub 流程处理。
- [ ] 所有 live 写动作仍要求审批、dry-run、日志、live 开关。
- [ ] 旧数据、演示数据不得触发真实审批或执行。
- [ ] 审批消息必须包含 `approval_id`、数据新鲜度和 dry-run 摘要；无 `approval_id` 的同意/拒绝不得执行。
- [ ] 生产环境有回滚方案和备份路径。

## 飞书最终回复准入规则

生产飞书不只检查工具输入，也要检查最终回复正文。Gateway 或 S3 消息中台在发送文本、文件、artifact、多维表格或文档前，都必须先执行等价于 `feishu-output-gate` 的安全判断。

必须拦截：

- 源码、`SKILL.md`、`references` 五件套、`runtime_commands.md`、完整 schema/rules/examples。
- 配置、env、角色表、数据库映射、DSN、真实 open_id/chat_id。
- CSV/XLSX/JSON/SQLite/SQL 原始数据和完整 runtime JSON。
- 订单明细、行级订单、订单流水、住客名单。
- `git stash/reset/clean/pull`、改文件、改配置、重启服务、安装模型/插件/应用等运维声明。
- 手动聊天补数绕过新鲜度、审批或数据来源校验。

允许输出：

- 脱敏摘要。
- 诊断结论。
- 证据日期。
- 数据新鲜度。
- 风险。
- 建议动作。
- 审批状态。

飞书多维表格和文档只能承载聚合摘要与受控报表字段，不得写入源码、配置、订单行级明细或数据库原始表。

## 经营快报和订单分析输出规则

- S2 经营快报必须按固定结构输出：结论、数据日期、核心指标、风险、建议动作、审批状态。
- 只有 `freshness_status=fresh` 且 `today_label_allowed=true` 时，才能称为“今日实时经营快报”。
- 混合今日房态、昨日经营指标、sample/hardcoded 指标时，只能称为“历史/演示复盘”或“今日房态摘要”。
- S16 实际进度优先使用订单业务日期或入住时间统计，不得使用数据上传时间。
- S17 只输出聚合订单分析
