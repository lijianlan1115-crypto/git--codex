# S15 销售基准线规则

## 算法来源
- 对应节点：N010 / S15 销售基准线。
- 对应 Agent：A2。
- 对应 BP：P0。
- 源文件索引：`references/source/source_manifest.yaml`。
- 字段契约：`contracts/node_io_contract.yaml`。
- runtime 规则：`runtime/algorithm_rules/baseline_rules.yaml`。
- 阈值、分时节点和人工基准线审核开关必须以 runtime 规则文件为准；skill 只负责加载、解释和输出，不自行发明阈值。
- 如果上述 source、contract 或 runtime 规则文件缺失，必须输出 `data_gap` 或 `config_gap`，不得把缺失配置伪装成 confirmed 规则。

## 输入分层
- hard_required：`hotel_id`、`data_business_date`。缺失时必须阻断主链路，返回 `missing_fields`，`confidence=low`。
- soft_required：`historical_same_period`、`demand_index`、`available_rooms`、`target_occupancy_rate`。缺失时可以继续生成预览或降级基准线，但必须输出 `data_gap`。
- optional：当前无额外可选字段。
- candidate：当前无候选字段；候选字段不得用于 live 判断。
- blocked_for_live：`demo_data`、`sample_data`、`stale`、`missing_date`。这些口径只能用于诊断、预览或 dry-run，不得创建正式审批或进入 live 执行。

## 字段兼容关系
- 当前 runtime 主字段是 `target_orders`，下游接受字段 `target_room_nights` 必须由它同步得到。
- 当前 runtime 主字段是 `hourly_curve`，下游接受字段 `hourly_target_curve` 必须由它同步得到。
- 当前 runtime 主字段 `confidence`、`source_confidence`、`decision_confidence` 可解释为 `baseline_confidence` 的依据。
- 当前 runtime 字段 `historical_same_weekday` 可作为 `historical_same_weekday_curve` 的历史依据；如果不是曲线结构，只能作为历史证据说明。
- `target_orders`、`hourly_curve`、`progress_checkpoints` 不得删除，S16 当前仍依赖这些字段。

## 判断逻辑
1. 优先执行 DataGate：检查 hard_required、字段质量、业务日期和数据新鲜度。
2. 加载历史目标、近期趋势、日期类型和市场上下文。
3. 优先采用 runtime `baseline` 输出，不在 skill 中重写确定性计算。
4. 目标来源优先级：`sales_baseline.target_orders` > `daily_metrics.room_nights` > 保守默认基准线。
5. 小时曲线来源优先级：`sales_baseline.hourly_curve` > runtime 默认锚点曲线。
6. 计算或解释 `daily_target`、`target_orders`、`target_room_nights`、`hourly_curve` 和分时节点。
7. 节点目标必须包含 12 点、16 点、20 点、23 点；S16 判断进度时优先用节点目标，不只看全天目标。
8. 对基准来源进行版本标记；人工录入、候选字段和降级字段必须显式标注，不得伪装成 confirmed。
9. 需求指数、需求等级、业务日历和节假日只作为目标解释和修正依据，不直接触发调价。
10. 调休上班日不能按普通周末高需求处理。
11. 月指标、本月累计、本年累计只能作为对比证据，不得折算成今日目标。
12. 缺历史销售基准线但有日指标时，`baseline_confidence` 不得高于 `medium`。
13. 缺销售基准线、日指标和可信日期时，`baseline_confidence` 必须为 `low`，且标注演示/降级口径。
14. 若 `freshness_status` 不是 `fresh`，只能输出历史或演示基准，不能称为实时今日目标。
15. 输出路由给 S16/S5/S3 时，必须保留 `freshness_status`、`data_business_date`、`baseline_confidence` 和 `downstream_usage`。

## 阈值和冲突处理
- 阈值与分级必须读取 `runtime/algorithm_rules/baseline_rules.yaml`。
- 当前基准线规则要求 `checkpoint_hours=[12,16,20,23]`。
- 当前基准线规则要求 `manual_baseline_requires_audit=true`；人工基准线必须保留审核标记，不得直接进入 live。
- 冲突处理顺序：DataGate > freshness > approval/live guard > price/budget guard > skill-specific threshold。

## 数据库来源
- 可读取 `database-query --template sales_baseline` 作为优先基准线输入。
- 可读取 `database-query --template daily_metrics`、`monthly_metrics` 或 `order_snapshot` 作为补充证据。
- 可读取 `database-query --template demand_context` 或 S4 输出作为需求指数和业务日历补充。
- MySQL `fact_daily_metrics` 日目标只采纳 `period_type=本日/今日/当天/当日` 等日口径。
- `fact_monthly_metrics` 和本月/本年累计只作对比证据，不得覆盖当天目标。
- 数据库来源必须只读，不允许写回外部业务库。

## 输出要求
- confirmed outputs：`daily_target`、`hourly_curve`。
- candidate outputs：`baseline_version`、`confidence`。
- 必须输出当前 runtime 字段：`target_orders`、`target_occupancy_rate`、`hourly_curve`、`progress_checkpoints`。
- 必须同步说明下游接受字段：`target_room_nights`、`hourly_target_curve`、`baseline_confidence`。
- 必须输出汇总对象 `sales_baseline`，用于下游读取日目标、目标出租率、目标来源和关键假设。
- 有历史同星期曲线时输出或说明 `historical_same_weekday_curve`；没有时必须说明历史不足。
- 必须输出 `target_basis_date`、`target_basis_type`、`source_confidence`、`decision_confidence`。
- 必须输出 `data_business_date`、`data_snapshot_time`、`freshness_status`、`business_status`、`today_label_allowed`。
- 必须输出 `progress_checkpoints`，至少包含 12、16、20、23 四个节点；当前 runtime 暂未直接输出 23 点时，skill 解释层必须说明该节点由全天目标收口得到。
- 必须输出 `downstream_usage`，说明 S16/S5/S9 是否可使用当前基准线。
- 缺字段时必须输出 `data_gaps` 或 `missing_fields`；预览或演示口径必须输出 `preview_only`。
- 输出要说明是否可作为 S16 今日进度判断依据，以及是否允许继续进入 S5/S6 调价链路。
- 飞书回复只输出脱敏摘要，不贴完整 runtime JSON、数据库原始表、配置或源码。

## 降级规则
- hard_required 缺失时返回 `missing_fields`，`confidence=low`，并阻断主链路。
- soft_required 缺失时继续生成预览或降级基准线，但必须输出 `data_gap` 和降级原因。
- 输入为 `demo_data`、`sample_data`、`stale` 或 `missing_date` 时，只能返回 `preview_only` 或 `dry_run`，不得创建正式审批或进入 live。
- 来源能力为 `read_only` 或 `manual` 时，只能输出建议、诊断或人工任务，不执行写动作。
- 历史同期、需求指数、可售房量或目标出租率不足时，不得输出高置信度。

## 安全规则
- S15 不执行任何写动作。
- 低质量字段只能用于诊断、提示或 dry-run，不得用于真实执行。
- 输出必须带 `data_business_date`、`data_snapshot_time`、`freshness_status`。
- `sample_data`、`demo_data`、`stale`、`missing_date` 不能创建正式调价审批。
