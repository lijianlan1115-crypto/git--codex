# 数据字段字典与来源优先级

本文是酒店 OTA 数字员工的统一字段池和来源治理规则。各 skill 的 `rules.md` 不再重复维护大段字段清单，只保留本 skill 的触发、判断、降级和安全规则。

## 1. 来源优先级

当资料冲突时，按以下顺序解释：

1. 用户最新指令和安全底线。
2. `最后一次会议讨论.md`。
3. `项目进展与说明.md`。
4. `整体架构分析(1).xlsx`。
5. `00-功能逻辑总览.md`。
6. API 文档、接口样例、爬虫/RPA 字段。

API 文档只进入字段适配，不直接决定业务策略。业务策略以会议、功能蓝图、验收规则和杨总确认口径为准。

## 2. 字段放置规则

| 内容 | 放置位置 | 说明 |
| --- | --- | --- |
| 全局字段池 | 本文件 | 经营、订单、房态、价格、OTA、竞品、口碑等字段统一维护 |
| 统一输出字段 | `requirements/统一数据契约.md` | runtime/skill 间结构化字段 |
| 数据库表字段映射 | `/etc/hotel-ota-ai/database-source.json` | 生产私有配置，不提交真实 DSN 或私有映射 |
| API 字段适配 | `requirements/渠道API适配策略.md`、`skills/hotel-ota/_shared/channel-api-map.md` | 只做字段归一化，不写业务策略 |
| 单 skill 判断规则 | 各 skill 的 `references/rules.md` | 只写本 skill 怎么判断、怎么降级、怎么安全输出 |
| 飞书最终格式 | `requirements/飞书输出规范.md`、S3 模板 | 所有 skill 最终回复统一由 S3/模板治理 |

## 4. 字段可信度

| 来源 | `data_source_type` | 可用于正式审批 | 说明 |
| --- | --- | --- | --- |
| 只读 MySQL/PMS/API 今日数据 | `mysql_db` / `pms_api` / `ota_api` | 可以，但必须 `fresh/current` | 仍需 admin/owner 审批 |
| 受控人工上传 | `manual_upload` | 可作为候选，需审计摘要 | 不等于聊天手动补数 |
| 聊天手动输入 | `manual_chat` | 不可以 | 只能作为待核对线索 |
| RPA 抓取 | `rpa` | 视稳定性和审计而定 | 需记录时间、截图/来源、失败告警 |
| sample/demo | `sample_data` / `demo_data` | 不可以 | 仅演示和模板验证 |
| 旧数据 | `stale` / `historical_only` | 不可以 | 不得称为今日实时 |

## 5. 写入 skill rules 的边界

`rules.md` 应该写：

- 本 skill 触发场景。
- 本 skill 的判断逻辑。
- 本 skill 如何处理缺字段、旧数据、demo 数据。
- 本 skill 是否允许触发下游 skill。
- 本 skill 的安全底线。

`rules.md` 不应该写：

- 全量 API 字段。
- Excel 字段池。
- 数据库原始表全字段。
- 飞书最终排版细节。
- 大段重复的全局安全规则。

如确需引用字段，只写本 skill 的核心字段，并引用本文件。
