# S13 评论回复 规则

## 核心输入字段
- `review_content`
- `sentiment`
- `issue_tags`
- `severity`
- `reply_draft`
- `approved_by`

## 判断逻辑
1. 先分类再写回复
2. 涉及补偿、争议、隐私必须审批
3. 默认只生成草稿，不自动发布

## 角色权限判断
- `admin` 和 `owner` 可审批公开评论回复发布。
- `operator` 可生成回复草稿和发起审批。
- `frontdesk` 可反馈客诉事实，但不可生成公开发布动作。
- `guest` 不触发 S13。
- 公开发布必须有 `approval_id` 和 `admin/owner` 审批；涉及补偿、隐私、争议时必须升级。

## 可配置参数
- 蓝图中未最终确认的阈值标记为 `configurable`。
- 多源资料冲突时输出 `needs_business_confirm`，并采用更保守建议。
- API 未确认时字段质量为 `manual_required` 或 `inferred`。

## 异常处理
- 缺关键字段时先追问或降级为 sample/manual/RPA，不让 skill 失败退出。
- 低质量字段只能用于诊断、提示和 dry-run，不得用于真实执行。
- 原始 API 状态码必须先由 runtime 转成统一枚举后再解释。

## 安全规则
- 真实调价、房量、推广、评论发布必须审批。
- 所有写动作默认 `dry_run=true`。
- 必须记录请求摘要、响应码、失败原因和人工处理建议。
