# 调价建议算法

## 1. 功能名称

房型级调价建议。

## 2. 目标

按房型输出具体调价金额，解释为什么调价、什么时候调价、调到多少钱。

第一版只给建议，不自动执行。

## 3. 策略模式

系统至少支持两套算法：

- 满房派：优先保出租率，在底价以上尽量减少空置。
- RevPAR派：平衡均价和出租率，追求单房收益。

利润派后续可扩展，暂放在收益最大化方案中。

## 4. 输入数据

### 4.1 房型基础

- 房型ID。
- 房型名称。
- 房型总数。
- 当前可售库存。
- 房型剩余可售比例。
- 当前房价。
- 当前活动后到手价。
- OTA活动叠加后底价。
- 是否引流房型。
- 是否主力房型。
- 是否高价房型。
- 是否缺陷房型。

### 4.2 经营数据

- 当前出租率。
- 当前房型出租率。
- 当前订单量。
- 当前预订进度。
- 当前取消率。
- 当前曝光量。
- 当前浏览量。
- 当前支付转化率。
- 当前需求指数。
- OTA健康分。

### 4.3 时间数据

- 当前日期。
- 当前小时。
- 距离入住日天数。
- 是否当天。
- 是否未来1-3天。
- 是否未来4-7天。
- 是否未来8-21天。
- 是否节假日或特殊日期。

### 4.4 竞对数据

- 固定竞对池价格区间。
- 竞对综合价格指数。
- 竞对价格变化预警等级。

注意：竞对只做参考和预警，不直接触发调价。

## 5. 核心指标

```text
房型剩余可售比例 = 房型可售库存 / 房型总房数
价格安全距离 = 当前活动后到手价 - OTA活动叠加后底价
需求指数 = 来自需求指数算法
转化健康 = OTA健康诊断中的转化分
预订进度差 = 当前预订进度 - 历史同小时预订进度
竞对价格指数 = 本店当前到手价 / 竞对价格中位值
```

## 6. 满房派调价逻辑

### 6.1 核心原则

满房派优先级：

```text
底价安全 > 出租率 > 转化率 > 均价
```

课程依据线索：

- 先有量再有价。
- 白天跑量稳收益，晚上酌情提价提收益。
- 18点前出租率冲刺 60%+ 是满房关键观察点。
- 库存多时降价跑量，库存少时稳价或涨价。
- 淡季价格相对稳定，不大涨不大降。

### 6.2 满房派判断树

```text
1. 活动后到手价是否低于底价？
 是 -> 不允许降价，输出底价风险。
 否 -> 进入下一步。

2. 房型剩余可售比例是否高？
 是 -> 看需求指数和转化率。
 否 -> 看是否涨价或稳价。

3. 需求指数低且库存高？
 是 -> 建议降价或上短促活动。

4. 需求指数高且库存低？
 是 -> 建议涨价。

5. 转化率低？
 是 -> 先诊断价差、房型、取消政策、活动，不直接大幅降价。

6. 当天18点前出租率不足60%？
 是 -> 偏跑量，降价或活动建议优先。
 否 -> 可稳价或小幅涨价。
```

### 6.3 满房派调价幅度

第一版采用可配置阶梯。

```text
低谷 + 库存高：-10% 或 -20元
平淡 + 库存高：-5% 或 -10元
旺日 + 库存低：+5% 或 +20元
爆发 + 库存低：+10% 或 +50元
```

具体金额取两者之一：

```text
建议调整金额 = max(配置固定金额, 当前价 × 配置比例)
```

最终价格：

```text
建议价格 = 当前价格 + 建议调整金额
```

底价约束：

```text
建议活动后到手价 >= OTA活动叠加后底价 × 特殊日期系数
```

价格取整：

```text
建议价格 = 按5元或10元取整
```

## 7. RevPAR派调价逻辑

### 7.1 核心原则

RevPAR派优先级：

```text
RevPAR目标 > 出租率稳定 > 均价提升 > 库存安全
```

公式：

```text
RevPAR = ADR × 出租率
RevPAR = 房费收入 / 可售房间数
```

### 7.2 RevPAR目标

第一版支持三种目标来源：

- 用户手动输入目标 RevPAR。
- 历史同日期属性 RevPAR。
- 近30天 RevPAR × 目标提升系数。

### 7.3 RevPAR派判断树

```text
1. 当前预计RevPAR是否低于目标？
 是 -> 判断是出租率不足还是ADR不足。

2. 出租率不足且库存高？
 是 -> 降价、缩价差或上活动，提高成交。

3. ADR不足但出租率进度正常？
 是 -> 小幅涨价或关闭低价活动。

4. 出租率和ADR均正常？
 是 -> 稳价。

5. RevPAR高于目标且库存低？
 是 -> 涨价或保护库存。
```

### 7.4 RevPAR派计算

```text
预计ADR = 当前已售收入 / 当前已售间夜
预计出租率 = 当前已售间夜 / 可售房间数
预计RevPAR = 预计ADR × 预计出租率
RevPAR缺口 = 目标RevPAR - 预计RevPAR
```

调价方向：

```text
如果 RevPAR缺口 > 0 且 出租率低于目标 -> 降价或活动
如果 RevPAR缺口 > 0 且 出租率达标但ADR低 -> 涨价
如果 RevPAR缺口 <= 0 -> 稳价或小幅涨价
```

## 8. 时间窗口规则

### 8.1 当天

- 小时级更新。
- 白天优先跑量。
- 晚上根据库存和需求酌情涨价。
- 18点前出租率不足 60% 时，满房派偏降价或促销。

### 8.2 未来1-3天

- 看预订进度和需求指数。
- 如果未来几天预订好，逐步上调。
- 不建议单次大幅跳价，除非特殊日期或需求爆发。

### 8.3 未来4-7天

- 看日期属性、历史需求和早定进度。
- 以价格体系稳定为主。

### 8.4 未来8-21天

- 以远期房态完整可售为主。
- 避免过早大量关房。
- 早定/连住价格需具备竞争力。

## 10. 异常处理

- 到手价低于底价：禁止降价，只能建议提高流量、优化转化或调整活动。
- 转化率异常低：不直接大幅降价，先进入流量转化诊断。
- 竞对降价：只触发预警，结合需求、库存和转化再判断。
- 数据不足：输出“建议观察”，不输出具体金额或只输出保守金额。

### 13.2 算法修正

调价建议不应只依赖“当前价格 + 库存 + 需求指数”，还应加入价格来源和平台价格一致性校验。

新增前置校验：

```text
1. 房型是否可售？
 否 -> 不输出调价，输出开关房/库存检查建议。

2. 活动后到手价是否低于底价？
 是 -> 禁止降价，建议检查活动叠加。

3. 各平台预付价是否异常不一致？
 是 -> 先修正平台价差，再谈调价。

4. 是否需要价格审批？
 是 -> 输出建议价和审批原因，不输出执行动作。
```

修正后的调价金额计算：

```text
基础调整额 = 当前价 × 策略调整比例
库存修正系数 = 根据房型剩余可售比例设定
需求修正系数 = 根据需求指数等级设定
时间修正系数 = 根据当天/未来1-3天/未来4-7天/远期设定
特殊日期系数 = 用户配置

建议调整额 = 基础调整额 × 库存修正系数 × 需求修正系数 × 时间修正系数
建议价格 = 当前价 + 建议调整额
最终建议价格 = max(建议价格, 反推后的底价安全价格)
```

底价安全价格：

```text
反推后的底价安全价格 = OTA活动叠加后底价 / 当前活动折扣综合系数
```

### 13.3 逐字稿校验依据

逐字稿第2条确认：涨价要看高流量还是低流量，低流量时稳价或压价提出租率；大床涨价时双床稳价，反之亦然，不能所有房型同时涨价。

逐字稿第2条确认：18点前冲刺 60%+ 出租率是满房关键观察点，白天跑量，晚上酌情提价。

逐字稿第8条确认：库存多降价跑量，库存少稳价或涨价；涨价条件是库存少、有效销售时间长、流量和转化正常。

逐字稿第15条确认：稳定满房阶段需要 45-60 天价格体系相对稳定，不能因为短期满房就整体大改价格。

### 13.4 准确性与效率结论

调价算法方向正确，但必须增加“价格一致性、活动叠加、可售状态、价格审批”四个前置门槛。否则系统可能给出看似合理但实际不可执行的价格建议。

本算法应优先使用小时级数据运行，但远期价格建议可每日运行一次。
