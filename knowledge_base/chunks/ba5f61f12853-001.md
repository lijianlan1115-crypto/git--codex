# S15 销售基准线规则

## 核心输入字段
- `hotel_id`
- `business_date`
- `room_total`
- `historical_same_period`
- `demand_index`
- `operating_snapshot`
- `calendar_context`
- `demand_reasons`
- `occupancy_rate`
- `adr`
- `revpar`

兼容旧字段：
- `target_orders`：等价于 `target_room_nights`
- `hourly_curve`：等价于 `hourly_target_curve`
- `historical_same_weekday`：等价于 `historical_same_weekday_curve`
- `historical_same_date_type`：可作为 `historical_same_period` 的补充依据
- `holiday_history`：可作为 `calendar_context` 的补充依据

## 判断逻辑
1. 优先使用 `historical_same_period` 和 `historical_same_weekday_curve` 形成基础曲线。
2. 使用 `room_total` 将订单/间夜目标约束在可售房量范围内。
3. 使用 N006 的 `demand_index`、`calendar_context`、`demand_reasons` 修正目标强度。
4. 使用 N005 的 `operating_snapshot`、`occupancy_rate`、`adr`、`revpar` 判断当前经营状态是否偏离历史。
5. 输出全天目标 `target_room_nights`，并同步兼容输出 `target_orders`。
6. 输出 `hourly_target_curve`，并同步兼容输出 `hourly_curve`。
7. 节假日、调休上班、节前节后、寒暑假和本地事件只影响基准线修正，不直接触发调价。
8. 调休上班日不能按普通周末高需求处理。
9. 缺历史数据时输出降级基准线，`baseline_confidence` 不得高于 `medium`。
10. 历史、需求和经营快照均不足时，`baseline_confidence` 必须为 `low`，并在说明中标注“历史不足/降级口径”。

## 数据库来源
- 可读取 `database-query --template sales_baseline` 作为优先基准线输入。
- 可读取 `database-query --template daily_metrics`、`monthly_metrics` 或 `order_snapshot` 作为补充证据。
- MySQL `fact_daily_metrics` 日目标只采纳 `period_type=本日/今日/当天/当日` 等日口径。
- `fact_monthly_metrics` 和本月/本年累计只作对比证据，不得覆盖当天目标。
- 数据库来源必须只读，不允许写回外部业务库。

## 输出要求
- 必须输出 `sales_baseline`、`target_room_nights`、`hourly_target_curve`、`baseline_confidence`。
- 必须输出 `historical_same_weekday_curve`；没有历史数据时输出空数组或降级曲线，并说明原因。
- 必须保留兼容字段 `target_orders`、`hourly_curve`，直到 runtime 和下游节点完全迁移。
- 必须输出 `target_basis_date`、`target_basis_type`、`source_confidence`、`decision_confidence` 或对应置信度说明。
- 必须输出 `progress_checkpoints`，至少包含 12、16、20 三个节点。
- 输出要说明是否可作为 S16 今日进度判断、S5 收益动作建议、S9 峰谷判断和 S3 调价建议的依据。

## 安全规则
- S15 不执行任何写动作。
- 低质量字段只能用于诊断、提示或 dry-run，不得用于真实执行。
- 输出必须带 `data_business_date`、`data_snapshot_time`、`freshness_status`。
