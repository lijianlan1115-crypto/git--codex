# 需求指数与流量峰谷算法

## 1. 功能名称

需求指数与流量峰谷判断。

## 2. 目标

按房型、日期、小时计算 0-100 的需求指数，用于调价、推广、活动、库存管理和前台任务。

本功能不单独区分“市场流量峰谷”和“本酒店流量峰谷”，因为市场大盘数据难以稳定获取。第一版使用本酒店历史数据、本酒店当前预订进度、OTA漏斗数据和日期属性进行判断。

## 3. 输入数据

### 3.1 日期属性

- 日期。
- 星期几。
- 是否工作日。
- 是否周末。
- 是否节假日。
- 是否节前。
- 是否节中。
- 是否节后。
- 是否特殊活动日。
- 特殊日期系数。

### 3.2 房型数据

- 房型ID。
- 房型总房数。
- 当前可售库存。
- 房型剩余可售比例。
- 当前价格。
- 活动后到手价。

### 3.3 历史数据

- 历史同星期订单量。
- 历史同星期出租率。
- 历史同日期属性订单量。
- 历史同日期属性出租率。
- 历史同日期属性 ADR。
- 历史同日期属性 RevPAR。
- 历史同小时订单进度。

### 3.4 当前数据

- 当前曝光量。
- 当前浏览量。
- 当前支付订单量。
- 当前出租率。
- 当前预订进度。
- 当前取消率。
- 当前流失率。

## 4. 核心指标

```text
曝光浏览转化率 = 浏览量 / 曝光量
浏览支付转化率 = 支付订单量 / 浏览量
当前预订进度 = 当前已售间夜 / 预计目标间夜
库存压力 = 1 - 房型剩余可售比例
历史需求比 = 当前订单量 / 历史同属性订单量
```

## 5. 需求指数分级

```text
0-25：低谷
26-50：平淡
51-75：旺日
76-100：爆发
```

## 6. 计算方法

### 6.1 标准化函数

每个子指标先转换成 0-100 分。

```text
标准分 = min(100, max(0, 当前值 / 基准值 × 50))
```

解释：

- 当前值等于基准值时，得 50 分。
- 当前值高于基准值时，高于 50 分。
- 当前值低于基准值时，低于 50 分。

### 6.2 子分数

```text
历史需求分 = 标准化(历史同属性订单量, 近30天日均订单量)
当前进度分 = 标准化(当前小时累计订单量, 历史同小时累计订单量)
流量分 = 标准化(当前曝光量, 历史同小时曝光量)
转化分 = 标准化(当前浏览支付转化率, 历史同属性浏览支付转化率)
库存压力分 = (1 - 房型剩余可售比例) × 100
日期属性分 = 工作日/周末/节假日/特殊活动对应分值
```

### 6.3 第一版权重

```text
需求指数 =
历史需求分 × 20%
+ 当前进度分 × 25%
+ 流量分 × 15%
+ 转化分 × 15%
+ 库存压力分 × 15%
+ 日期属性分 × 10%
```

权重必须可配置。

## 7. 判断逻辑

### 7.1 低谷

条件：

```text
需求指数 <= 25
```

含义：

- 当前需求弱。
- 优先保出租率。
- 调价倾向：稳价或降价。
- 推广倾向：只有转化健康时才建议加推广。
- 活动倾向：可考虑限时特惠、今夜特价或缩小价差。

### 7.2 平淡

条件：

```text
26 <= 需求指数 <= 50
```

含义：

- 需求普通。
- 优先检查转化，再决定是否调价。
- 调价倾向：小幅调整。

### 7.3 旺日

条件：

```text
51 <= 需求指数 <= 75
```

含义：

- 需求较好。
- 如库存少且转化正常，可小幅涨价。
- 如库存多，先跑量，不急于涨价。

### 7.4 爆发

条件：

```text
需求指数 >= 76
```

含义：

- 需求强。
- 如库存快速下降，可涨价。
- 如部分房型库存极低，可保护库存或关闭低价活动。

```json
{
 "room_type": "双人电竞房",

 "demand_index": 68,

}
```

## 9. 异常处理

- 如果历史数据不足 14 天，使用酒店整体均值替代房型均值。
- 如果曝光或浏览为 0，不计算转化率，标记为数据不足。
- 如果特殊活动由人工录入，则特殊活动系数优先级高于普通日期属性。
- 如果当天价格大幅变动，需求指数需标记“受价格扰动影响”。

## 10. 可配置参数

- 历史窗口：7天、14天、30天。
- 日期属性分。
- 特殊日期系数。
- 各子指标权重。
- 需求等级阈值。
- 是否按房型单独计算。
- 是否允许人工修正需求指数。

### 11.2 算法修正

上一版把市场峰谷主要建立在酒店自身历史数据上。根据数据列表，系统可以直接读取“流量峰谷时段”“区域活动、行业热度监控”和“节假日信息”，因此需求指数应改成“外部需求 + 本店预订进度 + 本店流量转化 + 房型库存”的混合模型。

修正后的第一版公式：

```text
需求指数 =
日期环境分 × 20%
+ 区域热度分 × 15%
+ 历史同期分 × 15%
+ 当前预订进度分 × 20%
+ 当前流量分 × 10%
+ 当前转化分 × 10%
+ 房型库存压力分 × 10%
```

如果平台已经给出“流量峰谷时段”，则作为校准项：

```text
平台标记低谷 -> 需求指数最高不超过50，除非当前预订进度明显高于历史
平台标记高峰 -> 需求指数最低不低于50，除非当前转化和预订进度明显低于历史
```

### 11.3 逐字稿校验依据

逐字稿第1条确认：节假日前后、旺季收尾、暑期结束、开学季、中秋/国庆等日期属性会造成需求快速变化。

逐字稿第8条确认：动态定价首先看市场需求预测，再看竞对价格变化和自身预订增速、库存。

逐字稿第17条确认：前台和店长需要结合库存、流量特点、高流低转/低流高转来决定当天策略。

### 11.4 准确性与效率结论

该算法可以保留，但应把“日期环境分”和“区域热度分”前置，避免只用本店订单数据导致反应滞后。

第一版不建议引入天气，因为用户明确不关注天气，且数据列表中没有稳定天气字段。
