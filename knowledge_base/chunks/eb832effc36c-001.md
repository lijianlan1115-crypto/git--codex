# 协作开发与测试入门教程

## 1. 这份教程给谁看

这份文档给后续一起参与的人使用，包括：

- 业务协作者：帮忙核对酒店运营逻辑、调价规则、口碑处理、前台任务。
- skill 优化者：修改 17 个 OpenClaw skill 的触发、规则、样例和回复。
- runtime 开发者：维护稳定脚本、字段适配、dry-run、审批和日志。
- 测试协作者：按验收用例在本地或服务器上跑通命令和飞书话术。
- 部署协作者：把本地工程更新到阿里服务器 OpenClaw workspace。

本项目不是普通的单一代码仓库，而是“OpenClaw workspace + 17 个 skill + runtime + 多源业务资料 + 部署文档”的组合工程。

## 3. 项目目录怎么看

```text
openclaw-hotel-ota-ai/
 AGENTS.md OpenClaw 总控 Agent 规则
 TOOLS.md runtime 调用规则
 BOOTSTRAP.md OpenClaw 进入 workspace 后的读取顺序
 README.md 项目总览
 config/ OpenClaw 和环境变量样例
 cron/ 定时任务脚本
 examples/ 飞书测试话术
 requirements/ 需求、教程、验收、架构和适配文档
 runtime/ 稳定可执行脚本
 skills/hotel-ota/ 17 个酒店 OTA skill
```

核心分工：

- `SKILL.md`：让 OpenClaw 知道什么时候触发 skill，以及触发后先读什么。
- `references/`：让协作者和模型知道这个 skill 的输入、输出、规则、样例和命令。
- `runtime/`：让稳定逻辑可执行、可测试、可复用。
- `requirements/`：让业务、开发、测试、部署都有共同依据。

## 5. 一个 skill 应该怎么改

以 `s05-revenue-decision` 为例：

```text
skills/hotel-ota/s05-revenue-decision/
 SKILL.md
 references/
 input_schema.json
 output_schema.json
 rules.md
 examples.md
 runtime_commands.md
```

修改顺序：

1. 先读 `SKILL.md`，确认触发语和核心流程。
2. 再读 `references/rules.md`，确认来源依据和判断逻辑。
3. 如果新增字段，改 `input_schema.json`。
4. 如果新增输出结构，改 `output_schema.json`，但不要翻译标准 JSON 字段。
5. 如果新增话术或测试场景，改 `examples.md`。
6. 如果新增 runtime 命令，改 `runtime_commands.md` 和 `requirements/skill_specs.yaml`。

必须遵守：

- 不改 skill 目录名。
- 不改 frontmatter 的 `name`。
- 业务说明用中文。
- JSON 字段、CLI 参数、API 方法、环境变量保持英文。
- 涉及真实写动作时，必须保留审批和 dry-run。

## 6. 哪些逻辑放 skill，哪些放 runtime

放 skill / references：

- 中文业务解释。
- 缺失信息追问。
- 飞书回复样例。
- 业务判断树。
- 评论回复草稿。
- 多策略取舍说明。

放 runtime：

- API 请求构造、签名、状态码转换。
- 字段归一化。
- JSON 输入输出校验。
- 审批拦截。
- dry-run 动作生成。
- 日志脱敏。
- 价格底线、涨降幅、live 开关。
- 需求指数、OTA 健康、偏差诊断、调价建议的可复核计算。

一句话：模型负责理解和沟通，runtime 负责稳定和可验证。
