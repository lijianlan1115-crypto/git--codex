# 多源依据治理

## 1. 结论

本项目不使用单一资料作为全部判断来源。不同资料各司其职：

- 用户最新指令决定当前优先级和最终取舍。
- 安全底线决定是否允许执行。
- 会议资料决定项目目标、交付边界和时间节点。
- 功能逻辑蓝图决定业务算法和运营判断。
- Excel 架构资料决定系统分层、数据流和模块关系。
- API 文档决定字段适配、请求构造和能力边界。
- OpenClaw 文档和当前工程包决定技术落地方式。

## 2. 优先级

1. 用户最新指令。
2. 安全底线：审批、dry-run、live 开关、日志、脱敏。
3. `最后一次会议讨论.md`、`项目进展与说明.md`、`信息清单.md`。
4. `功能逻辑蓝图/功能逻辑蓝图/*.md`。
5. `整体架构分析(1).xlsx`。
6. Beyondh、美团、订单来了等 API 文档和本地 API 参考资料。
7. OpenClaw 官方文档和当前工程包兼容性。

## 3. 冲突处理

- 目标冲突：听用户最新指令。
- 业务规则冲突：采用保守口径，并标记 `needs_business_confirm`。
- API 字段冲突：进入 `requirements/统一数据契约.md`，不得直接污染 skill。
- 执行安全冲突：一律采用更严格规则。
- 蓝图阈值未确认：标记 `configurable`，不得硬编码为唯一真理。

## 4. 对 skill 的要求

- 每个 `SKILL.md` 只做触发、导航、核心流程和安全底线。
- 每个 skill 的规则文件只写业务判断、降级策略和安全规则，不写会议、Excel、蓝图或 API 出处。
- 来源依据只在 `requirements/` 的治理文档中维护，除非用户明确询问“依据从哪来”，否则不进入飞书业务 Agent 的常规执行上下文。
- 每个 skill 的 `references/input_schema.json` 和 `output_schema.json` 必须使用统一数据契约和统一输出 envelope。
- 每个 skill 的 `references/examples.md` 必须包含飞书输入、sample/manual/RPA 输入、runtime 输出和最终中文回复。

## 5. 对 runtime 的要求

- 共享稳定逻辑放在 `runtime/` 包。
- `runtime/hotel_ota_runtime.py` 继续作为唯一兼容入口。
- 原始 API 状态码、签名、字段映射、审批拦截、dry-run 和日志脱敏必须由 runtime 处理。
- skill 不直接解释原始 API 状态码，不绕过 runtime 执行写动作。
