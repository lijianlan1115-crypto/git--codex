# OTA健康诊断算法

## 1. 功能名称

OTA平台数据健康诊断。

## 2. 目标

站在酒店业主视角，用 5 档健康分判断酒店在 OTA 平台上的经营状态，并输出异常原因和建议动作。

健康诊断不只看单一指标，而是看“流量、转化、价格、库存、口碑、订单稳定性”的组合。

## 3. 输入数据

### 3.1 OTA漏斗数据

- 曝光量。
- 浏览量。
- 提交订单量。
- 支付订单量。
- 取消订单量。
- 成交间夜。
- 流失量。

### 3.2 经营数据

- 出租率。
- ADR。
- RevPAR。
- 订单量。
- 间夜量。
- 房型剩余可售比例。
- 远期可售状态。

### 3.3 价格数据

- 当前最低到手价。
- 各房型当前价。
- 活动后到手价。
- 底价。
- 竞对综合价格指数。
- 平台间价格差。

### 3.4 口碑数据

- 点评分。
- 点评量。
- 好评量。
- 中评量。
- 差评量。
- 近7天差评率。
- 主要评论问题分类。

### 3.5 对比基准

- 本店历史同期。
- 本店近7/14/30天。
- 固定竞对池均值或中位数。

## 4. 核心指标

```text
曝光浏览转化率 = 浏览量 / 曝光量
浏览支付转化率 = 支付订单量 / 浏览量
支付取消率 = 取消订单量 / 支付订单量
流失率 = 流失量 / 浏览量
订单稳定性 = 1 - 订单量变异系数
价格竞争指数 = 本店当前到手价 / 竞对价格中位值
差评率 = 差评量 / 点评量
库存健康度 = 可售房型数量 / 总房型数量
```

## 5. 健康五档

```text
5分：优秀。继续保持，可适度提价或扩大流量。
4分：健康。正常运营，维持策略。
3分：观察。有轻微问题，需要关注趋势。
2分：预警。需要采取调整动作。
1分：严重异常。需要立即干预。
```

## 6. 评分模型

第一版采用 100 分制，再映射到 5 档。

```text
OTA健康分 =
流量分 × 20%
+ 转化分 × 25%
+ 价格分 × 15%
+ 库存分 × 15%
+ 口碑分 × 15%
+ 稳定性分 × 10%
```

映射：

```text
85-100 -> 5分
70-84 -> 4分
55-69 -> 3分
40-54 -> 2分
0-39 -> 1分
```

## 7. 子分数计算

### 7.1 流量分

```text
流量分 = 50 × 当前曝光量 / 历史同属性曝光量
```

封顶 100，低于 0 则按 0。

如果竞对曝光或排名数据可用，则加入竞对对比；不可用则只用自身历史。

### 7.2 转化分

课程参考线：

- 浏览/曝光转化率保持 10-14%。
- 支付/下单转化率保持 8-12%。
- 使用付费流量时，浏览转化率短期低于 10% 可能正常。

```text
一级转化分 = 当前曝光浏览转化率 / 目标曝光浏览转化率 × 100
二级转化分 = 当前浏览支付转化率 / 目标浏览支付转化率 × 100
转化分 = 一级转化分 × 40% + 二级转化分 × 60%
```

### 7.3 价格分

```text
价格分 = 100 - abs(价格竞争指数 - 1) × 100
```

解释：

- 接近竞对价格中位值，价格分较高。
- 明显高于或低于竞对，都需要解释。
- 低价不一定健康，可能伤害价格体系。

### 7.4 库存分

扣分项：

- 主力房型无房。
- 引流房型无房。
- 远期3-21天大量关房。
- 保留房、维修房、线下预留房占比过高。
- 大床或双床某一类过早卖完导致夜间需求流失。

### 7.5 口碑分

```text
口碑分 = 点评分分数 × 50% + 差评率分数 × 30% + 评论增长分 × 20%
```

评论问题会进入评论模块进一步分类。

### 7.6 稳定性分

```text
订单变异系数 = 近7天订单标准差 / 近7天订单均值
稳定性分 = 100 - 订单变异系数 × 100
```

## 8. 判断逻辑

### 8.1 健康

条件：

```text
健康档位 >= 4
```

输出：

- 保持价格体系。
- 可在需求旺、库存少时小幅涨价。
- 可适度扩大流量。

### 8.2 观察

条件：

```text
健康档位 = 3
```

输出：

- 标注最弱指标。
- 建议观察 1-3 天。
- 避免频繁大幅改价。

### 8.3 预警

条件：

```text
健康档位 = 2
```

输出：

- 进入流量转化诊断。
- 检查活动、价差、库存、评论。
- 如连续 3 天预警，升级为严重异常。

### 8.4 严重异常

条件：

```text
健康档位 = 1
```

输出：

- 店长介入。
- 立即检查平台活动、价格、库存、竞对、差评。
- 生成前台执行清单和运营整改清单。


## 10. 异常处理

- 付费流量开启后，转化分需要标记“付费流量扰动期”。
- 数据缺失时，不输出总分，只输出可计算项。
- 平台间价格异常时，价格分直接进入预警。
- 远期房态大量关闭时，库存分直接扣分。

## 11. 可配置参数

- 各子分权重。
- 健康分映射阈值。
- 转化率目标值。
- 历史对比窗口。
- 竞对对比方式。
- 差评扣分规则。
- 库存异常阈值。

## 12. 数据列表适配校验

### 12.1 可直接使用的数据

来自 `数据列表.txt` 的可用字段：

- HOS总分及构成：服务质量、经营质量、产能水平。
- 商家经营分：装修、商品、内容、体验、评价。
- 同行竞争力评分。
- 同行均值：入住间夜、满房率、销售均价。
- 同行排名：曝光、订单量维度。
- 商圈份额。
- 目标达成度。
- 曝光量、浏览人数、支付转化率。
- 流失订单数、流失间夜、流失金额。
- 评价总得分、服务、设施、卫生、位置、差评率。
- 30/180/365天新增评价数。
- 房型当日是否可售、每日可售房量、每日总房量。
- 平台规则变动、OTA异常消息、异常消息。

### 12.2 算法修正

OTA健康诊断应优先接入平台官方评分，因为这些评分本身已经代表平台对门店的权重判断。

修正后的评分模型：

```text
OTA健康分 =
平台官方健康分 × 25%
+ 流量转化分 × 25%
+ 经营结果分 × 15%
+ 价格竞争分 × 10%
+ 库存可售分 × 10%
+ 口碑分 × 10%
+ 异常风险分 × 5%
```

平台官方健康分：

```text
平台官方健康分 = HOS总分 × 60% + 商家经营分 × 40%
```

经营结果分：

```text
经营结果分 = 入住间夜表现 × 40% + 满房率表现 × 30% + 销售均价表现 × 30%
```

异常风险分：

```text
无异常消息 = 100
存在普通异常 = 60
存在OTA异常或平台规则风险 = 30
存在处罚/限流 = 0
```

### 12.3 逐字稿校验依据

逐字稿第1条确认：订单下降要按市场原因、平台原因、自身运营原因排查，平台处罚、跨平台补贴、平台自动改价都会造成异常。

逐字稿第15条确认：渠道运营稳定需要浏览/曝光转化率、支付/下单转化率、最终转化率、给房量、日均订单量、平台间价格无异常。

逐字稿第17条确认：前台需每2小时同步库存和流量转化数据，店长据此定策略。

### 12.4 准确性与效率结论

上一版 OTA健康诊断偏“自建评分”，现在应改为“平台官方评分 + 自建漏斗评分”的组合。这样更贴近 OTA 真实推流逻辑，也更容易向酒店业主解释。
